FASE 11 – CORREÇÃO DEFINITIVA DO BOTÃO "CRIAR CONTA" E DA INTERFACE DE REGISTRO
Data: 20/04/2026 (madrugada)
================================================================================

Contexto:
- O botão "Criar Conta" na tela de login, embora configurado, não executava ação alguma.

- Tentativas anteriores de correção no entergame.lua resultaram em chamadas a arquivos
  inexistentes (create_account.lua) ou caminhos incorretos devido à resolução relativa
  do dofile dentro do módulo client_entergame.
- Descoberta de um módulo avançado (CreateAccount.lua/createAccount.otui) que exigiria
  múltiplos endpoints adicionais no Login Server, sendo postergado.

Ações Realizadas:
1. Criação de um novo script de registro (registro.lua) para evitar conflitos com
   versões cacheadas de create_account.lua.
2. Implementação de uma interface de registro baseada em arquivo .otui
   (register_window.otui) para contornar limitações da API de criação dinâmica de widgets.
3. Posicionamento inicial dos arquivos na raiz do OTClient, seguido de realocação para
   modules/client_entergame/ a fim de resolver problemas de resolução de caminho do
   g_ui.displayUI.
4. Substituição do evento onClick do botão no entergame.lua para chamar dofile('/registro.lua')
   ou dofile('registro.lua') com caminho absoluto (usando '/').
5. Limpeza de cache do OTClient (pasta %APPDATA%\MCR\cache) para forçar o recarregamento
   dos scripts modificados.
6. Validação e ajuste de métodos incompatíveis no registro.lua:
   - Substituição de setSize(300,220) por setWidth(300) e setHeight(220).
   - Remoção de setModal(true) e window:center() em favor de propriedades OTML
     (modal: true, center: true) no arquivo .otui.

Problemas Encontrados:
- O OTClient Redemption não suporta os métodos setModal, center e setSize com dois
  argumentos inteiros em janelas criadas programaticamente.
- Caminhos relativos no dofile e g_ui.displayUI variavam conforme o contexto de execução
  (módulo vs. raiz), causando "file not found".
- Conflito entre a configuração do botão via scheduleEvent (antiga) e a criação
  programática definitiva no entergame.lua, resultando na persistência do evento errado.

Soluções Aplicadas:
- Adoção de um arquivo .otui dedicado (register_window.otui) com todas as propriedades
  de layout definidas declarativamente (modal, center, anchors), carregado via
  g_ui.displayUI('register_window.otui') a partir do registro.lua.
- Utilização de caminho absoluto com barra inicial ('/registro.lua') no dofile para
  garantir a localização correta independentemente do módulo.
- Remoção do bloco scheduleEvent antigo no entergame.lua, mantendo apenas a criação
  programática do botão com o evento atualizado.
- Posicionamento final dos arquivos registro.lua e register_window.otui na pasta
  modules/client_entergame/ para alinhamento com o contexto de carregamento do módulo.

Resultado:
- O botão "Criar Conta" agora dispara corretamente a abertura da janela de registro.
- A janela é exibida centralizada, modal e com todos os campos ancorados adequadamente.
- A criação de contas via POST /register é realizada com sucesso, utilizando hash SHA1
  compatível com o Canary.
- O fluxo de criação de conta 100% no cliente está plenamente operacional e estável.

LIÇÕES APRENDIDAS (COMPILADO)
================================================================================

- Arquivos .otmod usam OTML, que exige indentação com espaços (não tabs) e não aceita chaves {}.
- Métodos como setModal, setClosable, setAttribute não são garantidos no OTClient Redemption.
- A criação programática de widgets com g_ui.createWidget é portável, mas o uso de .otui com ancoragem oferece melhor UX.
- Codificação ISO-8859-1 é necessária para acentuação correta em scripts Lua no OTClient.
- O campo 'characters' na resposta JSON do login deve ser array vazio ([]), nunca null.
- As portas do Canary devem ser exatamente: status 7171, login 7172, game 7173.
- O hash de senha deve ser consistente entre Login Server e Canary.
- No PowerShell, 'curl' é alias de Invoke-WebRequest; usar curl.exe ou Invoke-RestMethod para testes.
- Para depurar panics no Login Server, adicionar recover() em main.go e redirecionar saída com Tee-Object.
- O schema do banco de dados do Canary 3.4.1 pode diferir do esperado pelo Login Server; verificar colunas das tabelas players e accounts.
- O OTClient Redemption não suporta setModal, center e setSize com dois argumentos; prefira propriedades OTML.
- Caminhos relativos no dofile dentro de módulos podem causar "file not found"; use caminho absoluto com '/'.
- Para garantir acentuação em mensagens do backend, salve arquivos Go como UTF-8 com BOM e defina charset=utf-8 nos headers.

FASE 18 – VALIDAÇÃO DA ENTRADA NO JOGO E CORREÇÃO DO ERRO 10060
Data: 20/04/2026 (noite)
================================================================================

Contexto:
- Um personagem de teste foi inserido manualmente na tabela `players` do banco
  `BancoServer` com posição e town válidos.
- Ao tentar entrar no jogo com esse personagem, o cliente exibia "Conectado no
  servidor do jogo..." seguido da mensagem "Sua conexão foi perdida. (ERRO 10060)".

Ações Realizadas:
1. Teste de conectividade com `Test-NetConnection 127.0.0.1 -Port 7173`, que confirmou
   que a porta estava acessível (TcpTestSucceeded = True).
2. Verificação do `config.lua` do Canary:
   - A porta `gameProtocolPort` estava configurada como `7173` (correto).
   - O IP de bind estava como `ip = "127.0.0.1"`.
3. Criação de regra de firewall no Windows para permitir tráfego TCP na porta 7173.
4. Alteração do IP de bind no `config.lua` para `ip = "0.0.0.0"` e reinício do Canary.
5. Apesar das correções, o erro 10060 persistiu.

Problemas Encontrados:
- O erro 10060 continuava mesmo com a porta `7173` respondendo a testes externos.
- Análise da resposta JSON do Login Server revelou que o campo `externalport` do mundo
  "MCR" estava sendo enviado como `7172` (porta de login) em vez de `7173` (porta de jogo).
- O cliente OTClient utilizava essa porta para conectar ao ProtocolGame, resultando em
  timeout (10060).

Soluções Aplicadas:

FASE 19 – CONFIGURAÇÃO DE CACHE LOCAL PARA O OTCLIENT
Data: 20/04/2026 (continuação)
================================================================================

Contexto:
- O OTClient Redemption, por padrão, armazena arquivos de cache e configurações
  no diretório `%APPDATA%\<nome_do_cliente>` (ex.: `C:\Users\<Usuário>\AppData\Roaming\MCR`).
- Essa localização pode dificultar a portabilidade do cliente (execução via pendrive,
  múltiplas instâncias com configurações isoladas) e a limpeza manual de cache durante
  o desenvolvimento.
- Desejava-se que todos os dados de cache fossem mantidos em uma subpasta `cache` na
  própria raiz do OTClient, tornando a instalação autossuficiente e facilitando
  testes com diferentes versões.

Ações Realizadas:
1. Criação manual da pasta `cache` no diretório raiz do OTClient.
2. Modificação do arquivo `init.lua` (raiz do cliente) para alterar a função de
   definição do diretório de escrita do usuário.
3. Substituição da linha:
   `g_resources.setupUserWriteDir(('%s/'):format(g_app.getCompactName()))`
   por:
   `g_resources.setWriteDir(g_resources.getWorkDir() .. 'cache')`
4. Teste de inicialização do OTClient para confirmar a criação automática de
   subdiretórios de cache (minimap, screenshots, etc.) dentro da nova pasta local.

Problemas Encontrados:
- Nenhum. A função `setWriteDir` aceita um caminho absoluto ou relativo ao diretório
  de trabalho, e `g_resources.getWorkDir()` retorna corretamente o caminho da raiz
  do executável do OTClient.
- Observação: o uso de `setWriteDir` em vez de `setupUserWriteDir` implica que o
  cliente não tentará criar o diretório baseado no nome compacto do aplicativo; cabe
  ao desenvolvedor garantir que a pasta `cache` exista (caso contrário, o cliente
  poderá falhar ao tentar escrever). A criação manual da pasta foi realizada como
  precaução.

Soluções Aplicadas:
- Criação prévia da pasta `cache` na raiz.
- Uso de `g_resources.getWorkDir() .. 'cache'` para montar o caminho absoluto,
  garantindo compatibilidade com qualquer local de execução.

Resultado:
- O OTClient agora armazena todo o cache (mapa, configurações de UI, dados de login
  salvos) dentro da pasta `cache` local, ao lado do executável.
- A portabilidade do cliente foi aprimorada, permitindo executar múltiplas cópias
  independentes a partir de diretórios distintos.
- O processo de limpeza de cache durante o desenvolvimento pode ser feito simplesmente
  excluindo a pasta `cache` e reiniciando o cliente.

- (criação manual) pasta `cache/` na raiz do OTClient

     evitando a complexidade de conversão de encoding em todas as camadas.

Data: 22/04/2026 (madrugada/manhã)
================================================================================

CONTEXTO:
Tentativas de implementar conversão ISO‑8859‑1 ↔ UTF‑8 no servidor e no cliente
mostraram-se complexas e propensas a erros. A exibição do nome do personagem
continuava corrompida ("JoÃ£o") mesmo com dados corretos no banco.

AÇÕES REALIZADAS:
1. **Investigação aprofundada:** Confirmado que o OTClient Redemption renderiza nomes
   de criatura como Latin‑1, ignorando configurações de encoding.
2. **Mudança de estratégia:** Em vez de lutar contra o encoding, adotou-se uma
   **lista branca de caracteres permitidos**, baseada nos caracteres que comprovadamente
   não causam corrupção visual.
3. **Implementação da validação:** Criada a função `isCharacterAllowed(byte)` que
   verifica cada byte da string do nome, bloqueando explicitamente `ã` (0xE3), `õ` (0xF5),
   `í` (0xED), entre outros problemáticos, e permitindo todos os ASCII e acentos comuns
   como `á`, `é`, `ç`, `â`, etc.

RESULTADO:
- Nomes como "João" ou "María" são rejeitados com uma mensagem clara.
- Nomes como "Jose", "Coração" (com `ç`) e "Águia" são aceitos e exibidos perfeitamente.
- A experiência do usuário permanece estável e previsível, sem surpresas de encoding.
