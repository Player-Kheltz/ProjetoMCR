1. VISÃO GERAL E OBJETIVOS
O Projeto MCR é um servidor de Tibia independente e profundamente customizado, originado do código‑fonte opentibiabr/canary. Distingue‑se de distribuições genéricas por sua filosofia de design focada em imersão narrativa em um mundo de fantasia original, praticidade para o jogador e utilização máxima dos recursos modernos oferecidos pelo cliente OTClient e pela Engine Canary.

Tema Central: “Uma Aventura Fantasiosa” – O MCR não replica locações ou histórias do Tibia oficial. Ele se passa em um universo próprio, onde a imaginação é o limite: reinos flutuantes, florestas vivas, cidades goblin industriais, dragões filósofos, maldições temporais e artefatos que desafiam as leis da física. Tudo é possível dentro do escopo técnico do projeto. A narrativa prioriza a relação do jogador com o mundo e a sensação de descoberta constante.

2.2. Servidor de Login (Login Server)
O Login Server é um componente separado e essencial para o Protocolo 15. Ele é responsável por autenticar as contas dos jogadores, fornecer a lista de personagens e, de acordo com a filosofia do Projeto MCR, também gerenciar a criação de novas contas diretamente através do cliente.

Item	Configuração / Recomendação
Repositório	Utilizar o repositório oficial: https://github.com/opentibiabr/login-server
Tecnologia	Desenvolvido em Go, oferecendo alta performance e baixo consumo de recursos.
Compilação/Execução	Baixar o binário mais recente da seção "Releases" do repositório para seu sistema operacional ou compilar via go build.
Banco de Dados	O Login Server deve ser configurado para acessar o mesmo banco de dados utilizado pelo Canary (BancoServer). Ele gerencia as tabelas de contas, personagens e autenticação.
Extensão de Funcionalidade (Criação de Conta via Cliente)	Foi implementado um endpoint REST /register (em src/api/register.go) que recebe dados de criação de conta (nome, e‑mail, senha), valida unicidade e força, gera um hash SHA1 da senha e insere o registro na tabela accounts. O hash SHA1 foi escolhido por praticidade e compatibilidade imediata com a configuração padrão do Canary, eliminando complexidades desnecessárias. O sistema aceita todos os caracteres válidos da língua portuguesa (acentos, cedilha, etc.) sem causar quebras no banco, no servidor ou no cliente.
Correção da Lista de Personagens	O endpoint /login.php (em src/api/login.go) foi ajustado para garantir que o campo characters seja retornado como um array vazio ([]) quando não houver personagens, prevenindo erro de iteração no OTClient.
Compatibilidade Nome de Conta × E‑mail	O Canary (v3.4.1) autentica utilizando o campo email da tabela accounts. Para permitir que os jogadores façam login com o nome da conta (campo name), o Login Server foi modificado (src/api/login.go) para, durante a autenticação, consultar o banco de dados e obter o email correspondente ao account_name fornecido. Esse e‑mail é então enviado ao Canary no processo de login, garantindo compatibilidade sem necessidade de alteração no servidor de jogo.
Variáveis de Ambiente (Arquivo .env)	Criar um arquivo .env no mesmo diretório do executável do Login Server com as seguintes variáveis:
dotenv
# Configurações do Banco de Dados (apontando para o mesmo BancoServer do Canary)
MYSQL_DBNAME=BancoServer
MYSQL_HOST=127.0.0.1
MYSQL_PORT=3306
MYSQL_USER=root
MYSQL_PASS=suasenha

# Configurações do Login Server
LOGIN_IP=0.0.0.0
LOGIN_HTTP_PORT=8080

# Configurações do Servidor de Jogo (Canary)
SERVER_IP=127.0.0.1
SERVER_NAME=ProjetoMCR
SERVER_PORT=7173   # ⚠️ ATENÇÃO CRÍTICA: Deve corresponder exatamente ao gameProtocolPort do Canary (7173). Usar 7172 (porta de login) causará erro 10060.
SERVER_LOCATION=Brasil
VOCATIONS=1,2,3,4

Legenda das Variáveis:
- MYSQL_*: Credenciais do banco de dados. Devem ser as mesmas utilizadas pelo Canary.
- LOGIN_IP e LOGIN_HTTP_PORT: IP e porta onde o Login Server escutará requisições HTTP. A porta 8080 é a padrão.
- SERVER_IP, SERVER_NAME, SERVER_PORT: Informações do servidor de jogo enviadas ao cliente após login. **SERVER_PORT deve coincidir com gamePort do Canary (7173).**
- SERVER_LOCATION: Localização geográfica exibida no cliente.
- VOCATIONS: Lista de IDs das vocações disponíveis.

2.4.2. Notas Técnicas Relevantes (Extraídas do DevLog)
- Scripts Lua com strings acentuadas carregados via dofile devem ser salvos como ISO-8859-1.
- O OTClient Redemption não suporta os métodos setModal, center e setSize com dois argumentos; prefira propriedades OTML (modal: true, center: true) ou cálculo manual de posição.
- O campo characters na resposta JSON do /login deve ser um array vazio ([]), nunca null.
- As portas do Canary devem ser: status 7171, login 7172, game 7173.
- A variável SERVER_PORT no .env do Login Server deve apontar para a porta do ProtocolGame (7173). ⚠️ Erro comum: usar 7172 causa timeout (10060).
- Para evitar panics no Login Server, adicione recover() em main.go e verifique a compatibilidade do schema do banco (ex.: coluna lastlogin ausente no Canary 3.4.1).
- **OTUI Syntax:** Em arquivos .otui, use `text` (não `!text`) e evite `HorizontalLayout` se houver problemas de ancoragem. Prefira posicionamento com `anchors` e margens.
- **Sintaxe OTUI:** Arquivos .otui não aceitam comentários com -- e não utilizam chaves {} para agrupar propriedades. A indentação deve ser feita exclusivamente com espaços. Widgets devem ser declarados como filhos diretos da janela principal para que getChildById funcione corretamente.
- **Criação de Personagem:** O OTClient Redemption **não** possui módulo nativo `game_charactercreation`. A interface de criação de personagem deve ser implementada via extensão do `client_characterlist` ou módulo customizado.
- **Ajuste de alinhamento de texto:** A propriedade `text-offset: 0 1` pode ser usada em `TextEdit` para corrigir desalinhamentos verticais sutis do texto digitado.
- **Encoding de Nomes de Personagem:** O OTClient Redemption exibe nomes de criaturas como Latin‑1 (ISO‑8859‑1). Caracteres como 'ã', 'õ', 'í' podem causar corrupção visual. Recomenda‑se validar os nomes com uma lista branca de caracteres seguros até que uma solução definitiva de encoding seja implementada.


2.4.3. Configuração de Cache Local (Portátil)
Para tornar o OTClient autossuficiente e facilitar a limpeza de cache durante o desenvolvimento, todos os dados de cache (minimap, screenshots, configurações salvas) devem ser armazenados em uma subpasta `cache/` na raiz do cliente, em vez do diretório `%APPDATA%`.

**Implementação:**
- Crie manualmente a pasta `cache/` na raiz do OTClient.
- No arquivo `init.lua` (raiz do cliente), substitua a linha:
  `g_resources.setupUserWriteDir(('%s/'):format(g_app.getCompactName()))`
  por:
  `g_resources.setWriteDir(g_resources.getWorkDir() .. 'cache')`

**Resultado:** O cliente passa a ser portátil (pode ser executado de qualquer local) e o cache pode ser limpo simplesmente excluindo a pasta `cache/`.

2.4.4. Gerenciamento de Fontes e Ajustes de Texto no OTClient

O OTClient utiliza fontes bitmap pré‑renderizadas, localizadas na pasta `data/fonts/`.
Isso significa que o tamanho da fonte é intrínseco ao arquivo de fonte utilizado, e
não existe uma propriedade universal `font-size` como em CSS.

Para controlar o tamanho do texto, deve‑se utilizar a propriedade `font` nos widgets,
especificando o nome de uma fonte bitmap disponível no cliente (ex.: `font: verdana-11px-rounded`).

**Catálogo de Fontes Bitmap (`.otfont`) Recomendadas:**
| Nome da Fonte | Tamanho | Uso Sugerido |
| :--- | :--- | :--- |
| `cipsoftFont` | ~10px | Fonte padrão do Tibia, para uso geral. |
| `verdana-11px-rounded` | 11px | Textos de destaque, títulos. |
| `verdana-10px` | 10px | Corpo de texto, rótulos. |
| `small-9px` | 9px | Textos pequenos, status. |
| `sans-bold-16px` | 16px | Títulos principais. |

**Nota:** O OTClient Redemption também suporta fontes TrueType (`.ttf`) na pasta
`data/fonts/ttf/`, porém seu uso pode exigir configuração adicional. Recomenda‑se
testar a compatibilidade antes de adotá‑las.

**Propriedade `text-offset`:**
A propriedade `text-offset` aceita dois valores inteiros (ex.: `text-offset: 0 1`) e
permite um ajuste fino do posicionamento do texto dentro de widgets como `TextEdit`
e `Label`. É uma ferramenta valiosa para correções de alinhamento visual, deslocando
o texto horizontal e verticalmente em pixels.

**Nota sobre `font-scale`:**
Versões mais recentes do OTClient (a partir da v3.1 com PR #485) introduziram o suporte
a `font-scale`, uma configuração global que escala todas as fontes proporcionalmente.
No entanto, a versão Release 4.0 (13/01/2026) utilizada no Projeto MCR não possui
esse recurso de forma estável, sendo a escolha de fontes bitmap o método recomendado.

2.6. Fluxo de Autenticação do Protocolo 15 (Criação de Conta Integrada)
O fluxo completo de um novo jogador no Projeto MCR ocorre inteiramente dentro do cliente OTClient, conforme descrito a seguir.

Primeiro Acesso – Criação de Conta:
1. O jogador abre o OTClient e, na tela inicial, clica no botão "Criar Conta" (integrado em entergame.lua).
2. O script registro.lua é executado, exibindo um formulário onde o jogador insere: nome de conta desejado, e‑mail, senha e confirmação de senha.
3. Após validação local (campos preenchidos, senhas coincidentes), o cliente envia uma requisição HTTP POST para o endpoint /register do Login Server (http://127.0.0.1:8080/register).
4. O Login Server valida os dados (unicidade do nome/e‑mail, força da senha), gera um hash SHA1 da senha, insere a nova conta no banco de dados BancoServer e retorna uma resposta de sucesso (ou erro descritivo).
5. O cliente exibe uma notificação visual confirmando a criação da conta.

Login:
6. O jogador insere suas credenciais na tela de login do OTClient.
7. O OTClient envia uma requisição HTTP POST para o Login Server (http://127.0.0.1:8080/login).
8. O Login Server autentica as credenciais consultando o banco de dados BancoServer e verificando o hash SHA1. Internamente, traduz o nome da conta para o e‑mail associado antes de comunicar‑se com o Canary.

Seleção de Personagem:
9. Em caso de sucesso, o Login Server retorna ao cliente a lista de personagens daquela conta. A lista é garantidamente um array JSON ([] quando vazia).
10. O cliente exibe a interface de lista de personagens (módulo client_characterlist).

**Observação sobre Criação de Personagem:**
A interface de criação de personagem **ainda não está implementada** no cliente. A lista de personagens vazia é exibida corretamente, mas o botão "Criar Personagem" não está presente no módulo client_characterlist. Testes de conectividade e entrada no jogo foram realizados com sucesso utilizando personagens inseridos manualmente no banco de dados. A implementação da interface de criação de personagem (seja estendendo client_characterlist ou utilizando um módulo customizado) é uma tarefa planejada e deve seguir a diretriz de experiência 100% no cliente.

Conexão ao Jogo:
11. O jogador seleciona um personagem existente. O cliente recebe do Login Server o endereço do servidor de jogo (Canary) para se conectar (configurado em SERVER_IP e SERVER_PORT=7173 no .env).
12. O OTClient então se conecta diretamente ao servidor Canary na porta gamePort (7173) e inicia a sessão de jogo.

Este fluxo garante que 100% da jornada do jogador, do primeiro acesso ao gameplay, ocorra dentro do cliente OTClient, sem qualquer necessidade de interação com um site externo.

2.7. Considerações de Segurança
Senhas: O projeto utiliza SHA1 para hash de senhas, garantindo compatibilidade imediata e simplificada entre Login Server e Canary. A robustez do sistema de criação de contas e login é a prioridade.

Conexões: Utilize sempre senhas fortes para o banco de dados e mantenha o software do servidor e do Login Server atualizados.

Firewall: Configure o firewall para permitir apenas as portas necessárias:
- 8080 (TCP) para o Login Server (acesso público HTTP).
- 7172 (TCP) para comunicação interna do Login Server com o Canary.
- 7173 (TCP) para o servidor de jogo (Canary), acesso público para os jogadores.
- 3306 (TCP) para o banco de dados (não deve ser exposto publicamente).

Criação de Conta via Cliente: Implementar rate limiting por IP e, futuramente, validação de e‑mail e captcha (ex.: hCaptcha) no endpoint /register para prevenir abusos e criação massiva de contas falsas.

2.8. Diretrizes para Experiência Exclusivamente no Cliente

2.9. Criação de Personagem – "Salão dos Destinos" (Protótipo Funcional)

A criação de personagem no Projeto MCR é realizada através de uma experiência narrativa
imersiva denominada **"Salão dos Destinos"**. Atualmente em fase de protótipo funcional,
o sistema permite que novos jogadores criem seus heróis diretamente no cliente OTClient,
sem necessidade de ferramentas externas.

Fluxo Atual (Protótipo):
1. O jogador faz login com uma conta que contém o personagem **"Alma"** (criado
   manualmente para testes). **Atualmente, "Alma" não é um personagem dummy global;
   está acessível apenas na conta onde foi criado.**
2. Ao entrar no jogo com "Alma", o personagem aparece em uma área isolada do mapa
   ("Salão dos Destinos") e é saudado pelo NPC **"Oráculo dos Caminhos"**.
3. Através de um diálogo imersivo no chat, o Oráculo guia o jogador na escolha de um
   nome e de uma vocação (Guerreiro, Arqueiro, Mago, Druida).
4. Após confirmação, o servidor valida os dados, cria o novo personagem na tabela `players`
   e desconecta o personagem "Alma".
5. O jogador é redirecionado à tela de seleção de personagens, onde o novo herói já
   aparece listado.

Validação de Nomes (Encoding):
Devido a limitações conhecidas do OTClient Redemption com a renderização de caracteres
acentuados em nomes de criaturas, o protótipo adota uma **lista branca de caracteres
permitidos**. São aceitos todos os caracteres ASCII imprimíveis (letras, números,
pontuação) e a maioria dos acentos comuns (ex.: á, é, ó, ç, â, ê, ô). Caracteres que
causam corrupção visual (ex.: ã, õ, í) são bloqueados com uma mensagem de erro clara.

Limitações Atuais (Protótipo):
- **O personagem "Alma" não é um dummy global.** Apenas a conta que o contém pode
  acessar o Salão dos Destinos. É necessário evoluir para um mecanismo que permita a
  qualquer nova conta iniciar o fluxo de criação.
- **Não há botão "Criar Personagem" na interface de seleção.** O acesso é feito
  entrando manualmente com "Alma".
- A lista branca de caracteres impede o uso de alguns acentos comuns em português.

Visão de Futuro:
O NPC Oráculo foi projetado para evoluir para um **"Administrador de Contas"** completo,
integrado ao cliente, oferecendo funcionalidades como:
- Alteração de senha.
- Recuperação de conta (token por e‑mail).
- Exclusão de personagens.

2.10. Criação e Posicionamento de NPCs no Mundo (Spawn)

O Canary utiliza o sistema Revscriptsys para scripts de NPC, que são arquivos .lua independentes e auto‑registráveis. No entanto, para que um NPC apareça no mapa, ainda é necessário declará-lo no arquivo de spawn do mundo. Esta seção documenta o processo completo e obrigatório para dar vida a um novo NPC.

2.10.1. Script do NPC (Revscriptsys)

O arquivo Lua do NPC deve ser colocado em data/npc/scripts/ e seguir a estrutura padrão (como canary.lua). Exemplo:

lua
local npcName = "MeuNovoNpc"
local npcType = Game.createNpcType(npcName)
local npcConfig = { name = npcName, /* ... configurações de outfit, loja, etc. ... */ }
-- ... lógica do NPC ...
npcType:register(npcConfig)
Nota: O nome definido em npcConfig.name DEVE corresponder exatamente ao atributo name no arquivo de spawn XML.

2.10.2. Arquivo de Spawn do Mundo (MCR-npc.xml)

O arquivo de spawn de NPCs está localizado em data/world/MCR-npc.xml. Ele contém uma lista de todos os NPCs que devem ser carregados automaticamente ao iniciar o servidor.

Estrutura do Arquivo:

xml
<?xml version="1.0"?>
<npcs>
    <!-- Exemplo de spawn para um NPC -->
    <npc centerx="666" centery="663" centerz="15" radius="6">
        <npc name="NomeExatoDoNPC" x="0" y="0" z="0" spawntime="60" />
    </npc>
</npcs>
Atributos Explicados:

<npcs>: Tag raiz obrigatória.

<npc> (container): Define uma área de spawn.

centerx, centery, centerz: Coordenadas do centro da área de spawn no mapa.

radius: Raio em tiles dentro do qual o NPC pode aparecer aleatoriamente (útil para criaturas, mas para NPCs geralmente usa-se radius="1" e coordenadas exatas).

<npc> (entidade): Define o NPC em si.

name: CRÍTICO. Deve ser idêntico ao npcConfig.name definido no script Lua.

x, y, z: Coordenadas relativas ao center do container. Para posicionamento exato, use x="0" y="0".

spawntime: Tempo em segundos para o NPC reaparecer se for removido (ex.: morto). Para NPCs, use um valor alto como 60 ou -1 para que nunca desapareça.

2.10.3. Procedimento Obrigatório para Adicionar um Novo NPC

Sempre que um novo NPC for criado, execute estes passos na ordem correta:

Crie o script Lua em data/npc/scripts/seu_npc.lua, definindo um npcConfig.name único e memorável.

Edite o arquivo data/world/MCR-npc.xml e adicione uma nova entrada <npc> com o name exato do script, posicionando-o nas coordenadas desejadas do mapa.

Reinicie o servidor Canary ou utilize o comando /reload npcs (se disponível) para aplicar as mudanças.

Checklist de Verificação:

O nome no script Lua (npcConfig.name) corresponde exatamente (case‑sensitive) ao atributo name no XML?

O arquivo Lua está salvo como UTF‑8 com BOM (para acentuação correta no servidor)?

As coordenadas no XML (centerx, centery, centerz) são válidas e estão em um tile acessível do mapa?

O servidor foi reiniciado após as alterações?

⚠️ Aviso de Segurança: Nunca delete o arquivo MCR-npc.xml nem o modifique sem backup. Uma tag mal formada pode impedir o carregamento de todos os NPCs do mundo.

4.3.5. Biblioteca de Suporte a Diálogos (NPC Utils) – Implementação de Referência

Para garantir que todos os NPCs do Projeto MCR ofereçam uma experiência de diálogo rica e consistente, é obrigatório utilizar ou adaptar a seguinte biblioteca de funções auxiliares. Este código deve residir em data/npc/lib/npc_utils.lua e ser carregado por todos os scripts de NPC via dofile('data/npc/lib/npc_utils.lua').

lua
--[[
    Projeto: MCR
    Módulo: Biblioteca de Suporte a Diálogos (NPC Utils)
    Arquivo: data/npc/lib/npc_utils.lua
    Descrição: Fornece funções padronizadas para reconhecimento flexível de intenções,
               personalização de pronomes/tratamento e variação de respostas.
--]]

NpcUtils = {}

-- Constantes de sexo (conforme definidas no Canary)
local PLAYERSEX_FEMALE = 1
local PLAYERSEX_MALE   = 0

--[[
    Verifica se a mensagem do jogador contém qualquer um dos sinônimos fornecidos.
    A verificação é case‑insensitive e busca a palavra inteira para evitar falsos positivos.

    @param mensagem string A mensagem digitada pelo jogador.
    @param listaSinonimos table Uma tabela de strings contendo os sinônimos.
    @return boolean True se a mensagem corresponder a pelo menos um sinônimo.
]]
function NpcUtils.correspondeAcao(mensagem, listaSinonimos)
    mensagem = mensagem:lower()
    for _, sinonimo in ipairs(listaSinonimos) do
        -- Padrão que busca a palavra inteira, cercada por espaços, pontuação ou início/fim da string
        local padrao = '[%p%c%z%s]' .. sinonimo:lower() .. '[%p%c%z%s]'
        if mensagem:find(padrao) or mensagem:find('^' .. sinonimo:lower() .. '$') or mensagem:find('^' .. sinonimo:lower() .. '[%p%c%z%s]') or mensagem:find('[%p%c%z%s]' .. sinonimo:lower() .. '$') then
            return true
        end
    end
    return false
end

--[[
    Retorna uma tabela com pronomes, artigos e vocativos personalizados de acordo com
    o sexo e a vocação do jogador. Essencial para uma narrativa imersiva e inclusiva.

    @param player userdata O objeto Player do jogador.
    @return table Tabela com os campos: artigo, pronome, possessivo, tratamento, bemVindo, sufixoVoc, vocativo.
]]
function NpcUtils.getTratamento(player)
    local sexo = player:getSex()
    local vocacao = player:getVocation():getName()
    local tr = {}

    if sexo == PLAYERSEX_FEMALE then
        tr.artigo = "a"; tr.pronome = "ela"; tr.possessivo = "sua"
        tr.tratamento = "senhorita"; tr.bemVindo = "bem‑vinda"; tr.sufixoVoc = "a"
    else
        tr.artigo = "o"; tr.pronome = "ele"; tr.possessivo = "seu"
        tr.tratamento = "senhor"; tr.bemVindo = "bem‑vindo"; tr.sufixoVoc = ""
    end

    -- Vocativos fantásticos de acordo com a vocação (padrão MCR)
    local vocativos = {
        Knight   = "guardião" .. tr.sufixoVoc .. " de armadura reluzente",
        Paladin  = "atirador" .. tr.sufixoVoc .. " de flechas sagradas",
        Sorcerer = "tecelão" .. tr.sufixoVoc .. " dos véus arcanos",
        Druid    = "guardiã" .. tr.sufixoVoc .. " dos ciclos naturais"
    }
    tr.vocativo = vocativos[vocacao] or "aventureir" .. tr.sufixoVoc .. " dos reinos distantes"
    
    return tr
end

--[[
    Seleciona aleatoriamente uma frase de uma lista e substitui o placeholder %s
    pelo nome do jogador, garantindo variação nas interações.

    @param listaFrases table Lista de strings com placeholders %s.
    @param player userdata O jogador para o qual a frase será personalizada.
    @return string A frase escolhida e formatada.
]]
function NpcUtils.escolherFraseVariada(listaFrases, player)
    local nome = player:getName()
    local frase = listaFrases[math.random(#listaFrases)]
    return frase:format(nome)
end

5. SISTEMA DE QUESTS HÍBRIDO (SQH) – CHAT IMERSIVO COM AÇÕES CONTEXTUAIS
Escopo: Evolução do antigo SQT para um modelo que combina narrativa rica no chat com interatividade moderna (links clicáveis, HUD persistente, notificações toast).

5.1. Visão Geral do Subsistema
Este subsistema redefine a interação de missões no Projeto MCR. O chat continua sendo o palco principal da narrativa, garantindo que o jogador leia e se conecte com os NPCs. Entretanto, para eliminar a fricção de ter que digitar repetidamente palavras‑chave, as ações mais importantes são apresentadas como links clicáveis diretamente no texto. Além disso, um HUD de Missões (Quest Tracker) fornece um lembrete visual constante do progresso, e notificações toast entregam feedback imediato.

Objetivo Principal: Oferecer uma experiência narrativa profunda, sem sacrificar a conveniência que os jogadores modernos esperam.

5.2. Convenção de Marcação: Colchetes [ ] para Links Clicáveis
Para diferenciar os elementos interativos do texto narrativo, utilizamos colchetes [ ]. O cliente OTClient será customizado para renderizar qualquer texto dentro de colchetes como um hyperlink clicável. Ao ser clicado, o cliente envia automaticamente o texto (sem os colchetes) como se o jogador o tivesse digitado.

Categoria	Formato	Exemplo Visual no Chat	Ação no Cliente
Ação (Verbo)	[verbo]	[aceitar]	Ao clicar, envia o comando aceitar.
Item de Quest	[quantidade Nome do Item]	[10 Peles de Lobo Etéreo]	Não é clicável (apenas destaque visual). Pode ser opcionalmente configurado para abrir uma descrição do item.
Local/NPC Alvo	[Nome do Local]	[Torre do Astrônomo Perdido]	Não é clicável por padrão, mas pode ser usado para abrir o mapa ou adicionar um marcador.

Nota: O uso de {chaves} fica reservado para destaque puramente visual, enquanto [colchetes] indica um elemento interativo.

5.3. Fluxograma de Interação Híbrida
Este é o fluxo padrão que todo NPC de quest deve seguir. Proíbe‑se o uso de player:showModalWindow() para progresso de missão.

1. Saudação Contextual: O NPC identifica o jogador e seu estado de missão.
   Exemplo: "Ah, [Jogador]! Vejo que retornou. Conseguiu encontrar as [10 Peles de Lobo Etéreo]?"

2. Reconhecimento de Intenção: O script escuta palavras‑chave específicas para disparar ações.
   Exemplo: Se o jogador digitar missão, tarefa, trabalho.
   Resposta do NPC: "Preciso que você vá até o [Lago dos Reflexos] e me traga [5 Lágrimas de Sereia]. O que me diz? [aceitar]?"

3. Confirmação Interativa (Chat com Links): O jogador pode:
   - Clicar no link [aceitar] no chat.
   - Digitar aceitar, sim ou qualquer sinônimo configurado.

4. Ação do Servidor: Seta storage de missão ativa e:
   - Envia um Extended Opcode para atualizar o HUD de Missões.
   - Opcionalmente, dispara uma notificação toast: "Missão iniciada: Lágrimas de Sereia".

5. Acompanhamento pelo HUD: Durante a missão, o jogador pode ver o progresso no canto da tela:
   [Missão] Lágrimas de Sereia
   * Coletar Lágrimas de Sereia: 2/5
   O HUD é atualizado automaticamente via Opcodes quando o jogador coleta os itens.

6. Entrega/Progresso: O jogador retorna ao NPC e diz entregar ou clica no link [entregar] no diálogo do NPC.

7. Validação do Servidor: Checa itens, remove, concede recompensa.

8. Sucesso: NPC envia mensagem narrativa com destaque para a recompensa:
   "Incrível! Tome seus [100 Fragmentos de Prata Lunar] e este [Talismã da Maré]."
   O servidor envia Opcodes para limpar o HUD da missão e exibir uma notificação toast de conclusão.

5.4. Implementação Técnica (Servidor e Cliente)
5.4.1. Marcação no Servidor (Canary - Lua)
Os diálogos dos NPCs devem utilizar colchetes para os termos clicáveis.
lua

A lógica de escuta (creatureSayCallback) permanece idêntica à do antigo SQT. O clique no link apenas envia a palavra (ex: aceitar) como se fosse digitada.

5.4.3. HUD de Missões (Quest Tracker)

-- Estrutura do HUD: um painel simples no canto superior direito
local trackerPanel = nil
local titleLabel = nil
local objectiveLabel = nil

function QuestTracker.update(data)
    if not trackerPanel then
        trackerPanel = g_ui.displayUI('quest_tracker.otui')
        titleLabel = trackerPanel:getChildById('title')
        objectiveLabel = trackerPanel:getChildById('objective')
    end
    titleLabel:setText(data.titulo)
    objectiveLabel:setText(string.format("%s: %d/%d", data.descricao, data.atual, data.total))
    trackerPanel:show()
end

function QuestTracker.clear()
    if trackerPanel then trackerPanel:hide() end
end

-- Registra listener para Extended Opcode
ProtocolGame.registerExtendedOpcode(OPCODE_QUEST_UPDATE, function(protocol, opcode, buffer)
    local data = json.decode(buffer)
    if data then
        QuestTracker.update(data)
    end
end)

5.5. Diretrizes de Qualidade para Quests Híbridas
- Proibição de Janelas Modais: O NPC de quest nunca deve chamar player:showModalWindow() para decisões de missão. Exceção: Loja, Banco e confirmações críticas.
- Uso Consistente de Colchetes: Termos clicáveis DEVEM estar entre colchetes [ ].
- HUD Sempre Visível: O Quest Tracker deve aparecer assim que o jogador tiver uma missão ativa e desaparecer ao concluí‑la.
- Feedback Imediato: Toda mudança de estado da missão deve gerar uma notificação toast e/ou som sutil.
- Respeito à Narrativa: Mesmo com links clicáveis, o diálogo do NPC deve permanecer rico e variado.

5.6. Lista de Verificação de Implementação (Checklist SQH)
Antes de marcar uma quest como "Concluída", verifique:
- Os diálogos do NPC usam colchetes para termos acionáveis: [aceitar], [entregar].
- O cliente foi configurado para renderizar [texto] como links clicáveis.
- Não há chamadas para player:showModalWindow() no fluxo da missão.
- Ao aceitar a missão, um Opcode é enviado para exibir/atualizar o HUD de Missões.
- Ao coletar itens, o progresso no HUD é atualizado em tempo real.
- Ao concluir a missão, o HUD é ocultado e uma notificação toast é exibida.
- A recompensa é entregue diretamente no inventário e anunciada no chat com destaque.

5.7. Exemplo Completo: Quest "O Lamento da Sereia"

Este exemplo demonstra a implementação completa de uma quest simples utilizando o Sistema de Quests Híbrido (SQH) e a biblioteca npc_utils.lua. Ele deve ser usado como modelo obrigatório para todas as novas quests.

5.7.1. Estrutura do NPC (data/npc/scripts/sereia_lunar.lua)

lua
-- Cabeçalho padrão MCR
local keywordHandler = KeywordHandler:new()
local npcHandler = NpcHandler:new(keywordHandler)
NpcSystem.parseParameters(npcHandler)

-- Carrega a biblioteca de utilidades OBRIGATÓRIA
dofile('data/npc/lib/npc_utils.lua')

-- Constantes da Quest (IDs de Storage reservados conforme Seção 6.6)
local STORAGE_QUEST_SEREIA = 50001
local STORAGE_LAGRIMAS_COLETADAS = 50002
local QUANTIDADE_NECESSARIA = 5
local ITEM_LAGRIMA_SEREIA = 11111 -- VERIFICAÇÃO PENDENTE: ID real do item

-- Opcode para comunicação com cliente (HUD/Toast)
local OPCODE_QUEST_UPDATE = 180 -- VERIFICAÇÃO PENDENTE: Confirmar faixa de opcodes livre

function creatureSayCallback(cid, type, msg)
    local player = Player(cid)
    if not player then return true end
    msg = msg:lower()
    
    -- Estado 1: Jogador nunca iniciou a quest
    if player:getStorageValue(STORAGE_QUEST_SEREIA) < 1 then
        if NpcUtils.correspondeAcao(msg, {"missao", "quest", "tarefa", "ajuda", "trabalho"}) then
            local tr = NpcUtils.getTratamento(player)
            local saudacao = NpcUtils.escolherFraseVariada(NpcUtils.frasesSaudacao, player)
            npcHandler:say(saudacao, cid)
            npcHandler:say("Preciso que colete para mim [5 Lágrimas de Sereia] no [Lago dos Reflexos]. Você [aceita] este desafio, " .. tr.artigo .. " " .. tr.vocativo .. "?", cid)
            npcHandler.topic[cid] = 1 -- Aguarda resposta 'aceitar'
        end
    -- Estado 2: Missão em andamento
    elseif player:getStorageValue(STORAGE_QUEST_SEREIA) == 1 then
        -- (Lógica de entrega, verificação de itens, conclusão...)
    end
    return true
end

-- Callback para processar 'aceitar' dito ou clicado via link
function onAcceptQuest(cid, player)
    if player:getStorageValue(STORAGE_QUEST_SEREIA) < 1 then
        player:setStorageValue(STORAGE_QUEST_SEREIA, 1)
        player:setStorageValue(STORAGE_LAGRIMAS_COLETADAS, 0)
        
        -- Envia atualização para o HUD do cliente
        local hudData = { titulo = "O Lamento da Sereia", descricao = "Coletar Lágrimas de Sereia", atual = 0, total = QUANTIDADE_NECESSARIA }
        player:sendExtendedOpcode(OPCODE_QUEST_UPDATE, json.encode(hudData))
        
        -- Feedback imediato no chat e toast
        npcHandler:say("Que os ventos lhe guiem! Retorne assim que tiver as lágrimas.", cid)
        -- (Enviar opcode de Toast seria feito aqui)
        npcHandler.topic[cid] = 0
    end
end

-- Registro da callback no handler
npcHandler:setCallback(CALLBACK_MESSAGE_DEFAULT, creatureSayCallback)
npcHandler:setCallback(CALLBACK_ON_ACCEPT, onAcceptQuest) -- Implementar lógica para vincular 'aceitar' à callback
npcHandler:addModule(FocusModule:new())

ID do Item ITEM_LAGRIMA_SEREIA: Deve ser substituído pelo ID real do item no jogo, após criação no mapa/items.xml.

Validação de Itens: A função que verifica se o jogador possui os itens no inventário e os remove deve ser implementada com player:getItemCount() e player:removeItem().

6. META‑INSTRUÇÕES PARA O DESENVOLVIMENTO DO PROJETO MCR
Esta seção consolida todas as diretrizes obrigatórias que regem a criação de conteúdo, codificação, segurança e garantia de qualidade no Projeto MCR. Todo desenvolvedor, colaborador ou assistente de IA deve seguir estritamente estas instruções.

6.1. Diretriz Geral de Verificação de Código
⚠️ OBRIGATÓRIO ANTES DE GERAR QUALQUER CÓDIGO

Sempre que um script fizer referência a qualquer elemento externo ou dependência lógica, execute o seguinte procedimento de verificação:

1. Identificação de Elementos Verificáveis:
   - IDs numéricos (itens, outfits, montarias, efeitos, storages, opcodes)
   - Nomes de funções/métodos (ex.: player:sendExtendedOpcode, Game.createItem)
   - Campos de configuração (ex.: OTCFEATURES.modalWindows, entradas em config.lua)
   - Constantes e enums (ex.: TALKTYPE_SAY, PLAYERSEX_FEMALE)
   - Parâmetros de funções e seus valores esperados (ex.: true/false, strings específicas)
   - Estruturas de dados retornadas por APIs (ex.: getVocation():getName())

2. Pesquisa de Validação:
   Consulte fontes confiáveis e atualizadas:
   - Documentação Oficial do OpenTibiaBR: https://docs.opentibiabr.com
   - Tibia Wiki (em inglês ou português) para IDs de itens e mecânicas padrão
   - Repositórios oficiais: opentibiabr/canary (tag v3.4.1), opentibiabr/otclient, opentibiabr/login-server
   - Fóruns da comunidade: otland.net

3. Validação:
   - Confirmado: Utilize o valor/nome/função conforme documentado.
   - Não Confirmado: Adicione um comentário no código:
     lua
     -- VERIFICAÇÃO PENDENTE: Não foi possível confirmar se [elemento] é válido no contexto atual.
     -- Fonte consultada: [URL ou referência]. Necessário teste manual.
     E inclua o item no Relatório de Verificações Pendentes ao final da resposta.

4. Verificação de Coerência Lógica:
   - O método chamado é compatível com o objeto? (ex.: player:sendExtendedOpcode é válido; item:sendExtendedOpcode não)
   - Os parâmetros passados correspondem à assinatura esperada?
   - O valor booleano de configuração (true/false) está alinhado com o comportamento desejado e com as capacidades do servidor/cliente?
   - A lógica condicional (if/else) cobre todos os cenários possíveis?

6.2. Padrão de Codificação e Comentários
- Idioma dos Comentários: Português do Brasil (pt‑BR). Todos os comentários em arquivos .cpp, .h e .lua devem ser escritos em português claro e correto.
- Cabeçalho de Arquivo: Todo novo arquivo Lua ou módulo deve conter um cabeçalho descritivo:
  lua
  --[[
      Projeto: MCR
      Módulo: [Nome do Módulo]
      Autor: [Nome ou "Equipe MCR"]
      Data: [DD/MM/AAAA]
      Descrição: [Explicação concisa do propósito do arquivo.]
  --]]
- Nomes de Funções e Variáveis: Utilize camelCase para variáveis e funções locais, e PascalCase para funções globais/bibliotecas. Exemplo: local playerName, function QuestSystem.updateProgress().
- Evite Código Mágico: Números "mágicos" (valores literais sem explicação) são proibidos. Atribua‑os a constantes nomeadas com comentários. Exemplo: local MAX_PLAYER_LEVEL = 500.

6.3. Segurança e Validação de Dados
- Sanitização de Entrada do Jogador: Ao processar texto digitado pelo jogador (ex.: em onCreatureSay), sempre converta para minúsculas (msg:lower()) e remova espaços extras antes de aplicar find ou comparações.
- Validação de Dados do Cliente: Nunca confie cegamente nos dados recebidos do cliente. Sempre valide parâmetros de Extended Opcodes no lado do servidor. Verifique se os IDs recebidos existem, se as quantidades são válidas e se o jogador tem permissão para a ação.
- Prevenção de Injeção de SQL: Embora o Canary utilize consultas parametrizadas, ao escrever queries manuais (ex.: db.query), nunca concatene strings diretamente. Use sempre placeholders (%s, %d).
- Proteção de Loops Infinitos: Em loops while ou repeat, garanta que haja uma condição de saída clara e um limite máximo de iterações para evitar travamentos do servidor.

6.4. Performance e Otimização
- Extended Opcodes: O tamanho máximo de um pacote Extended Opcode é de 65KB. Evite enviar grandes volumes de dados de uma só vez. Para dados extensos, utilize múltiplos pacotes ou um sistema de paginação.
- Acesso a Storage: Evite chamadas excessivas a player:getStorageValue() dentro de loops. Armazene o valor em uma variável local sempre que possível.
- Iteração sobre Jogadores: Ao iterar sobre Game.getPlayers(), esteja ciente de que isso pode ser custoso em servidores com muitos jogadores online. Utilize eventos específicos ou sistemas de cache quando aplicável.

6.5. Consistência de Design e Narrativa (Tema Fantasia)
- Tom de Voz dos NPCs: Mantenha a coerência com o mundo fantasioso. Um mercador anão deve ter um tom diferente de um elfo ancião. Consulte a biblioteca npc_utils.lua para funções de personalização.
- Uso de Janelas Modais: Estritamente limitado a situações críticas que demandam confirmação explícita e não devem ser ignoradas (ex.: reset de atributos, ações destrutivas irreversíveis), além de Loja e Banco. Jamais utilize janelas modais para aceitar ou entregar missões.
- Destaque de Texto: Utilize colchetes [ ] para termos interativos (clicáveis). Chaves { } podem ser usadas para destaque visual não‑interativo.

6.6. Gerenciamento de Armazenamento (Storages e KVPs)
- Faixas de Storage: Para evitar conflitos, defina e documente faixas exclusivas para cada sistema do MCR. Sugestão inicial:
  - 50000 – 50999: Missões e Progresso de História.
  - 51000 – 51999: Habilidades Especiais e Talentos.
  - 52000 – 52999: Configurações do Jogador (UI, preferências).
- Dados Complexos: Para armazenar tabelas ou múltiplos valores, utilize json.encode() e json.decode() em uma única storage, ao invés de múltiplas storages separadas.
- Limpeza: Crie scripts de limpeza (/limparstorages) para remover dados de teste e permitir que jogadores reiniciem missões, se for uma funcionalidade desejada.
Faixas de Extended Opcodes (Comunicação Cliente‑Servidor):
Para evitar conflitos com opcodes nativos do Canary e de outras customizações, todas as comunicações customizadas do Projeto MCR devem utilizar opcodes na faixa de 160 a 255.

160 – 179: Reservado para sistema de criação de personagem ("Salão dos Destinos").

180 – 199: Reservado para o Sistema de Quests Híbrido (SQH) – atualizações de HUD, notificações toast.

200 – 219: Reservado para o futuro "Administrador de Contas".

220 – 255: Livre para outros sistemas.
Registro: Todo opcode utilizado deve ser documentado no MCR_DevLog.txt e na seção de Opcodes deste documento.

6.7. Testes e Depuração
- Modo Debug: Inclua uma variável global MCR_DEBUG = true no servidor que, quando ativada, imprime logs detalhados no console sobre o funcionamento dos sistemas.
- Comandos de Admin: Para cada sistema complexo (ex.: missões), crie comandos de administrador (ex.: /setquestprogress [id] [valor]) para facilitar os testes.
- Tratamento de Erros: Utilize pcall em chamadas de json.encode/decode para capturar erros de formatação e evitar quebras no script.

6.8. Roteiro Prático de Desenvolvimento de Conteúdo
Ao criar qualquer novo conteúdo (NPC, missão, sistema) para o mundo de fantasia do MCR, siga este fluxo de trabalho obrigatório:

1. Definição da Experiência do Jogador
   - Pergunta‑chave: Como esta interação pode contribuir para a imersão no mundo e se alinhar ao tema "Aventura Fantasiosa"?
   - Ação: Consulte o Catálogo de Funcionalidades (Seção 4.1) e o Princípio de Escolha de Ferramenta (Seção 4.2). Priorize o chat narrativo com links clicáveis e o HUD de missões. Janelas modais apenas para Loja/Banco e confirmações críticas.

2. Planejamento da Lógica e Armazenamento
   - Storages: Defina chaves de storage numéricas. Aplique a Meta‑Instrução de Verificação de Código (Seção 6.1).
   - Dados Complexos: Para estruturas maiores, utilize JSON armazenado em uma única storage ou campos customizados na tabela players do banco BancoServer.

3. Codificação do Script (Lua)
   - Idioma: Comentários e strings em pt‑BR.
   - Comunicação: Use player:sendExtendedOpcode(codigo, json.encode(dados)) para notificar o cliente sobre atualizações de HUD e toasts.
   - Documentação: Adicione comentários de VERIFICAÇÃO PENDENTE onde aplicável.
   - Segurança: Aplique as validações da Meta‑Instrução 6.3.

4. Criação da Contraparte no Cliente
   - Módulo: Crie ou edite arquivos na pasta modules/ do OTClient.
   - Listener: Registre um onExtendedOpcode para os códigos definidos no servidor.
   - UI: Manipule widgets (.otui) para o HUD de Missões e Notificações Toast.
   - Console: Customize o game_console para renderizar [texto] como links clicáveis.
   - Idioma: Todas as strings da interface em pt‑BR.

5. Teste e Refinamento
   - Teste a interação em ambiente de desenvolvimento.
   - Verifique se a acentuação está correta em todas as mensagens.
   - Avalie a fluidez, a imersão narrativa e a segurança (Meta‑Instrução 6.7).

6.9. Protocolo de Revisão Tripla e Autorevisão Obrigatória
Antes de finalizar QUALQUER resposta relacionada ao projeto MCR — seja gerando código, sugerindo alterações ou explicando conceitos — o assistente ou desenvolvedor DEVE executar internamente o seguinte protocolo de revisão tripla, desenhado especificamente para as demandas do Projeto MCR. Além disso, a conclusão bem‑sucedida desse protocolo deve ser evidenciada na própria resposta, conforme descrito ao final desta seção.

6.9.1. Execução Interna da Revisão Tripla

6.10. Princípio de Experiência Exclusivamente no Cliente (Meta‑Instrução)
⚠️ REGRA FUNDAMENTAL DO PROJETO MCR

Nenhuma experiência essencial do jogador deve ocorrer fora do cliente OTClient. Este princípio é absoluto e orienta todas as decisões de arquitetura e implementação.

Escopo de Aplicação Obrigatória:
- Criação de Conta: Interface completa de registro deve existir dentro do cliente. O jogador não deve ser redirecionado a um site externo para criar sua conta.
- Login e Autenticação: O processo de login deve ser realizado inteiramente através da interface do cliente.
- Seleção e Criação de Personagens: Deve ser possível criar, selecionar e gerenciar personagens dentro do cliente.
- Recuperação de Senha: Mecanismo de solicitação de recuperação deve estar disponível no cliente. A redefinição pode envolver um link enviado por e‑mail (única exceção aceitável, pois valida a propriedade do e‑mail).
- Gerenciamento de Conta: Alteração de e‑mail, senha, e preferências deve ser possível via interface in‑game.
- Quests e Progresso: Toda a narrativa e mecânica de missões deve ser acessada e gerenciada dentro do cliente.
- Loja e Compras: Todas as transações da loja devem ser realizadas via módulo game_shop integrado ao cliente.
- Notícias e Atualizações: Informações do servidor devem ser exibidas no MOTD ou em painéis acessíveis in‑game.

Verificação de Possibilidade Técnica:
Caso uma funcionalidade desejada não exista nativamente no ecossistema Canary/OTClient, a equipe de desenvolvimento DEVE:
1. Pesquisar se a funcionalidade pode ser implementada via:
   - Extended Opcodes (comunicação cliente‑servidor).
   - Módulos Lua no OTClient (criação de interfaces customizadas).
   - Comunicação HTTP direta entre cliente e Login Server (para endpoints REST).
2. Documentar a abordagem proposta e sua viabilidade.
3. Nunca optar por uma solução que force o jogador a sair do cliente sem antes esgotar todas as possibilidades de implementação dentro do próprio cliente.

Exceções Permitidas:
- Confirmação de E‑mail: O envio de um e‑mail de confirmação para validar a propriedade do endereço de e‑mail é aceitável, pois é um mecanismo de segurança padrão da indústria. No entanto, a interface para solicitar o reenvio do e‑mail de confirmação deve estar disponível dentro do cliente.
- Downloads e Atualizações: O processo de download do cliente e de atualizações automáticas (patcher) pode ocorrer fora do jogo, mas deve ser integrado a um launcher que preceda a execução do OTClient.

6.11. Diretrizes para APIs de Autenticação Exclusivas do Cliente
Para suportar a criação de conta e o gerenciamento de perfil diretamente do OTClient, o Login Server deve expor os seguintes endpoints REST. Estes endpoints serão consumidos pelo script de criação de conta (registro.lua) e futuramente por módulos de gerenciamento de conta no cliente.

Endpoint	Método	Descrição	Validações Obrigatórias e Comportamento Esperado
/register	POST	Cria uma nova conta de jogador.	Entrada (JSON): { "account": "nomeUsuario", "email": "email@exemplo.com", "password": "senha123" }

Validações:
- account: 3 a 32 caracteres alfanuméricos (pode incluir acentos e cedilha, pois o banco está em utf8mb4). Deve ser único na tabela accounts.
- email: Formato de e‑mail válido (regex). Deve ser único na tabela accounts.
- password: Mínimo de 8 caracteres, contendo pelo menos uma letra e um número.
- Rate Limiting: Máximo de 5 tentativas por IP a cada 10 minutos.
- Captcha: Recomenda‑se a integração com hCaptcha para prevenir bots (a ser implementado).

Processamento:
1. Validar campos conforme acima.
2. Gerar hash SHA1 da senha.
3. Inserir registro na tabela accounts com password = hash, email = email, creation = NOW().
4. Retornar status HTTP 200 em caso de sucesso, ou 400/409 com mensagem de erro descritiva em pt‑BR.

Exemplo de resposta de sucesso: {"status":"success","message":"Conta criada com sucesso!"}
Exemplo de erro: {"status":"error","message":"O nome de conta já está em uso."}
/recover	POST	Solicita recuperação de senha.	Entrada (JSON): { "email": "email@exemplo.com" }

Validações:
- email cadastrado na tabela accounts.
- Rate limiting: Máximo 3 tentativas por e‑mail por hora.

Processamento:
1. Gerar token único (UUID) com validade de 1 hora.
2. Armazenar token em tabela account_recovery.
3. Enviar e‑mail para o endereço fornecido contendo link para redefinição (ex.: http://projetomcr.com/reset?token=UUID).
4. Retornar mensagem genérica de sucesso (para evitar enumeração de e‑mails).
/reset	POST	Redefine a senha usando token.	Entrada (JSON): { "token": "UUID", "new_password": "novaSenha123" }

Validações:
- token válido, não expirado e não utilizado.
- new_password atende aos critérios de força (mínimo 8 caracteres, letras e números).

Processamento:
1. Localizar conta associada ao token.
2. Atualizar password na tabela accounts com o hash SHA1 da nova senha.
3. Invalidar o token.
4. Retornar confirmação de sucesso.
/change_email	POST	Altera o e‑mail da conta.	Entrada (JSON): { "account": "nomeUsuario", "current_password": "senhaAtual", "new_email": "novo@exemplo.com" }

Validações:
- Autenticação via account e current_password (verificar hash SHA1).
- new_email válido e único no banco.

Processamento:
1. Autenticar o jogador.
2. Atualizar email na tabela accounts.
3. Enviar e‑mail de confirmação para o novo endereço (opcional, mas recomendado).
4. Retornar sucesso.
/change_password	POST	Altera a senha da conta (via interface in‑game).	Entrada (JSON): { "account": "nomeUsuario", "current_password": "senhaAtual", "new_password": "novaSenha123" }

Validações:
- Autenticação via senha atual.
- new_password atende aos critérios de força.

Processamento:
1. Autenticar.
2. Atualizar password com hash SHA1 da nova senha.
3. Retornar sucesso.

Observações de Segurança e Implementação:
- Todos os endpoints devem operar sobre HTTPS em ambiente de produção.
- As respostas devem incluir cabeçalhos CORS apropriados para permitir requisições originadas do OTClient (Origin: http://localhost ou domínio do cliente).
- O hash de senha utilizado é SHA1, aplicado diretamente na senha em texto plano (sem salt adicional, seguindo o padrão atual do Canary configurado com passwordType = "sha1"). Isso garante máxima compatibilidade entre Login Server e servidor de jogo.
- A tabela accounts utilizada pelo Login Server deve ser a mesma do Canary (BancoServer.accounts), com os campos name, password, email, etc.
- **Para garantir a correta exibição de caracteres acentuados nas respostas JSON, salve os arquivos fonte do Login Server (.go) com codificação UTF-8 com BOM e certifique-se de que o middleware ou handler defina o cabeçalho Content-Type: application/json; charset=utf-8. Evite escapes Unicode (\u00e7) – use os caracteres literalmente.**
- **Para garantir a exibição correta de mensagens acentuadas no OTClient sem depender da codificação do backend, adote o padrão de códigos de erro (`error_code`).** O servidor retorna um código padronizado (ex.: `"account_name_taken"`), e o cliente mapeia esse código para uma string em português definida localmente no script Lua (salvo como ISO‑8859‑1). Isso elimina definitivamente problemas de encoding como "jÃi estÃi".
