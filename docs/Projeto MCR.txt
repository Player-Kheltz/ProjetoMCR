1. VISÃO GERAL E OBJETIVOS
O Projeto MCR é um servidor de Tibia independente e profundamente customizado, originado
do código‑fonte opentibiabr/canary. Distingue‑se de distribuições genéricas por sua
filosofia de design focada em imersão narrativa em um mundo de fantasia original,
praticidade para o jogador e utilização máxima dos recursos modernos oferecidos pelo
cliente OTClient e pela Engine Canary.

Tema Central: “Uma Aventura Fantasiosa” – O MCR não replica locações ou histórias do
Tibia oficial. Ele se passa em um universo próprio, onde a imaginação é o limite:
reinos flutuantes, florestas vivas, cidades goblin industriais, dragões filósofos,
maldições temporais e artefatos que desafiam as leis da física. Tudo é possível dentro
do escopo técnico do projeto. A narrativa prioriza a relação do jogador com o mundo e
a sensação de descoberta constante.

1.1. Pilares do Projeto
Pilar	Descrição
Imersão Narrativa em pt‑BR	Todo o conteúdo textual é localizado em português do Brasil com correção gramatical e acentuação impecável. NPCs possuem personalidade, histórias próprias e reagem ao contexto do personagem (sexo, vocação, progresso em missões). A ambientação é fantasiosa, rica em detalhes e referências a eventos e locais únicos do mundo MCR.
Experiência de Jogo Moderna	Substituição de interações arcaicas por uma interface híbrida e intuitiva: o chat continua sendo o palco narrativo, mas agora com palavras-chave destacadas em cores para indicar ações; HUD persistente para rastreamento de missões; notificações toast para feedback; e janelas modais estritamente para interações administrativas e críticas.
Estabilidade e Performance	Utilização do OTClient (opentibiabr/otclient), reconhecido por seu motor de renderização otimizado e total compatibilidade com o protocolo 15.00 do Canary. O servidor Canary é modificado para garantir suporte pleno aos recursos do cliente.
Independência Criativa	O MCR é um hobby project com identidade própria. Embora herde a estrutura do Canary, todas as adições de conteúdo, sistemas e narrativas são originais, construindo um mundo de fantasia coeso e livre de amarras do material oficial.

1.2. Escopo de Customização – Alcance Máximo
O Projeto MCR não se limita a scripts ou configurações superficiais. Para alcançar a
experiência imersiva e 100% integrada ao cliente, **todos os componentes do ecossistema
são passíveis de modificação em nível de código‑fonte**.

Esta filosofia aplica-se integralmente a:
- Servidor de Jogo (Canary): Customizações em C++ e Lua para adicionar novos sistemas, modificar comportamentos nativos, ajustar a comunicação de rede e personalizar a identidade visual do console.
- Cliente (OTClient Redemption): Alterações no motor C++ e nos módulos Lua para implementar interfaces exclusivas (ex.: Account Manager "Alma"), adaptar a experiência de autenticação e, se necessário, modificar a renderização do chat.
- Servidor de Login (Login Server): Expansão da API REST em Go para suportar criação automática de contas e gerenciamento de personagens diretamente pelo cliente.
- Editor de Mapas (Remere's Map Editor): Ajustes para compatibilidade com extensões customizadas do protocolo 15.00 ou para facilitar a criação de conteúdo específico do mundo MCR.

Regra de Ouro: Se uma funcionalidade desejada não puder ser implementada apenas com scripts, o desenvolvedor deve avaliar e, se viável, modificar o código‑fonte correspondente, documentando a alteração no `MCR_DevLog.txt`.

2.2. Servidor de Login (Login Server)
O Login Server é um componente separado e essencial para o Protocolo 15. Ele é responsável por autenticar as contas dos jogadores, fornecer a lista de personagens e, de acordo com a filosofia do Projeto MCR, também gerenciar a criação automática de novas contas através do Account Manager "Alma".

Item	Configuração / Recomendação
Repositório	Utilizar o repositório oficial: https://github.com/opentibiabr/login-server
Tecnologia	Desenvolvido em Go, oferecendo alta performance e baixo consumo de recursos.
Compilação/Execução	Baixar o binário mais recente da seção "Releases" do repositório para seu sistema operacional ou compilar via go build.
Banco de Dados	O Login Server deve ser configurado para acessar o mesmo banco de dados utilizado pelo Canary (BancoServer). Ele gerencia as tabelas de contas, personagens e autenticação.
Extensão de Funcionalidade (Criação de Conta Automática)	Foi implementado um endpoint REST /register (em src/api/register.go) que recebe dados de criação de conta (nome de conta, senha), valida unicidade e força, gera um hash SHA1 da senha e insere o registro na tabela accounts. O campo e‑mail não é utilizado, sendo definido como uma string vazia ou um valor placeholder. O hash SHA1 foi escolhido por praticidade e compatibilidade imediata com a configuração padrão do Canary.
Correção da Lista de Personagens	O endpoint /login.php (em src/api/login.go) foi ajustado para garantir que o campo characters seja retornado como um array vazio ([]) quando não houver personagens, prevenindo erro de iteração no OTClient.
Compatibilidade Nome de Conta × E‑mail	O Canary (v3.4.1) autentica utilizando o campo email da tabela accounts. Para permitir que os jogadores façam login com o nome da conta (campo name), o Login Server foi modificado (src/api/login.go) para, durante a autenticação, consultar o banco de dados e obter o email associado (que é um placeholder). Esse valor é então enviado ao Canary no processo de login, garantindo compatibilidade sem necessidade de alteração no servidor de jogo.
Variáveis de Ambiente (Arquivo .env)	Criar um arquivo .env no mesmo diretório do executável do Login Server com as seguintes variáveis:
dotenv
# Configurações do Banco de Dados (apontando para o mesmo BancoServer do Canary)
MYSQL_DBNAME=BancoServer
MYSQL_HOST=127.0.0.1
MYSQL_PORT=3306
MYSQL_USER=root
MYSQL_PASS=<Senha> (não informada por questão de segurança, arquivo com acesso publico no Git)

# Configurações do Login Server
LOGIN_IP=0.0.0.0
LOGIN_HTTP_PORT=8080

# Configurações do Servidor de Jogo (Canary)
SERVER_IP=127.0.0.1
SERVER_NAME=ProjetoMCR
SERVER_PORT=7173   # ⚠️ ATENÇÃO CRÍTICA: Deve corresponder exatamente ao gameProtocolPort do Canary (7173). Usar 7172 (porta de login) causará erro 10060.
SERVER_LOCATION=Brasil
VOCATIONS=1,2,3,4

Legenda das Variáveis:
- MYSQL_*: Credenciais do banco de dados. Devem ser as mesmas utilizadas pelo Canary.
- LOGIN_IP e LOGIN_HTTP_PORT: IP e porta onde o Login Server escutará requisições HTTP. A porta 8080 é a padrão.
- SERVER_IP, SERVER_NAME, SERVER_PORT: Informações do servidor de jogo enviadas ao cliente após login. **SERVER_PORT deve coincidir com gamePort do Canary (7173).**
- SERVER_LOCATION: Localização geográfica exibida no cliente.
- VOCATIONS: Lista de IDs das vocações disponíveis.

**Nota:** O arquivo `.env` com as credenciais reais **não é versionado** (listado no `.gitignore`).
Um template de exemplo está disponível em `LoginServer/.env.example` com placeholders
para orientar a configuração local.

2.4.2. Notas Técnicas Relevantes (Extraídas do DevLog)
- Scripts Lua com strings acentuadas carregados via dofile devem ser salvos como ISO-8859-1.
- O OTClient Redemption não suporta os métodos setModal, center e setSize com dois argumentos; prefira propriedades OTML (modal: true, center: true) ou cálculo manual de posição.
- O campo characters na resposta JSON do /login deve ser um array vazio ([]), nunca null.
- As portas do Canary devem ser exatamente: status 7171, login 7172, game 7173.
- A variável SERVER_PORT no .env do Login Server deve apontar para a porta do ProtocolGame (7173). ⚠️ Erro comum: usar 7172 causa timeout (10060).
- Para evitar panics no Login Server, adicione recover() em main.go e verifique a compatibilidade do schema do banco (ex.: coluna lastlogin ausente no Canary 3.4.1).
- **OTUI Syntax:** Em arquivos .otui, use `text` (não `!text`) e evite `HorizontalLayout` se houver problemas de ancoragem. Prefira posicionamento com `anchors` e margens.
- **Sintaxe OTUI:** Arquivos .otui não aceitam comentários com -- e não utilizam chaves {} para agrupar propriedades. A indentação deve ser feita exclusivamente com espaços. Widgets devem ser declarados como filhos diretos da janela principal para que getChildById funcione corretamente.
- **Criação de Personagem:** O OTClient Redemption **não** possui módulo nativo `game_charactercreation`. A interface de criação de personagem é implementada via diálogo com o NPC Oráculo no "Salão dos Destinos".
- **Ajuste de alinhamento de texto:** A propriedade `text-offset: 0 1` pode ser usada em `TextEdit` para corrigir desalinhamentos verticais sutis do texto digitado.
- **Encoding de Nomes de Personagem:** O OTClient Redemption exibe nomes de criaturas como Latin‑1 (ISO‑8859‑1). Caracteres como 'ã', 'õ', 'í' podem causar corrupção visual. Recomenda‑se validar os nomes com uma lista branca de caracteres seguros até que uma solução definitiva de encoding seja implementada.
- **Destaque de Palavras no Chat:** Em vez de modificar o cliente para criar hyperlinks complexos, utiliza‑se o sistema nativo de cores do Canary/OTClient. O servidor envia mensagens com tags `{{#RRGGBB}}texto{{/}}` para destacar palavras‑chave (ex.: `{{#00BFFF}}aceitar{{/}}`), guiando o jogador de forma visual e intuitiva.

2.4.3. Configuração de Cache Local (Portátil)
Para tornar o OTClient autossuficiente e facilitar a limpeza de cache durante o desenvolvimento, todos os dados de cache (minimap, screenshots, configurações salvas) devem ser armazenados em uma subpasta `cache/` na raiz do cliente, em vez do diretório `%APPDATA%`.

**Implementação:**
- Crie manualmente a pasta `cache/` na raiz do OTClient.
- No arquivo `init.lua` (raiz do cliente), substitua a linha:
  `g_resources.setupUserWriteDir(('%s/'):format(g_app.getCompactName()))`
  por:
  `g_resources.setWriteDir(g_resources.getWorkDir() .. 'cache')`

**Resultado:** O cliente passa a ser portátil (pode ser executado de qualquer local) e o cache pode ser limpo simplesmente excluindo a pasta `cache/`.

2.4.4. Gerenciamento de Fontes e Ajustes de Texto no OTClient

O OTClient utiliza fontes bitmap pré‑renderizadas, localizadas na pasta `data/fonts/`.
Isso significa que o tamanho da fonte é intrínseco ao arquivo de fonte utilizado, e
não existe uma propriedade universal `font-size` como em CSS.

Para controlar o tamanho do texto, deve‑se utilizar a propriedade `font` nos widgets,
especificando o nome de uma fonte bitmap disponível no cliente (ex.: `font: verdana-11px-rounded`).

**Catálogo de Fontes Bitmap (`.otfont`) Recomendadas:**
| Nome da Fonte | Tamanho | Uso Sugerido |
| :--- | :--- | :--- |
| `cipsoftFont` | ~10px | Fonte padrão do Tibia, para uso geral. |
| `verdana-11px-rounded` | 11px | Textos de destaque, títulos. |
| `verdana-10px` | 10px | Corpo de texto, rótulos. |
| `small-9px` | 9px | Textos pequenos, status. |
| `sans-bold-16px` | 16px | Títulos principais. |

**Nota:** O OTClient Redemption também suporta fontes TrueType (`.ttf`) na pasta
`data/fonts/ttf/`, porém seu uso pode exigir configuração adicional. Recomenda‑se
testar a compatibilidade antes de adotá‑las.

**Propriedade `text-offset`:**
A propriedade `text-offset` aceita dois valores inteiros (ex.: `text-offset: 0 1`) e
permite um ajuste fino do posicionamento do texto dentro de widgets como `TextEdit`
e `Label`. É uma ferramenta valiosa para correções de alinhamento visual, deslocando
o texto horizontal e verticalmente em pixels.

**Nota sobre `font-scale`:**
Versões mais recentes do OTClient (a partir da v3.1 com PR #485) introduziram o suporte
a `font-scale`, uma configuração global que escala todas as fontes proporcionalmente.
No entanto, a versão Release 4.0 (13/01/2026) utilizada no Projeto MCR não possui
esse recurso de forma estável, sendo a escolha de fontes bitmap o método recomendado.

2.6. Fluxo de Autenticação e Criação de Conta (Account Manager "Alma")
O fluxo completo de um novo jogador no Projeto MCR ocorre inteiramente dentro do cliente OTClient. O sistema de Account Manager, inspirado nos OTServs antigos, elimina a necessidade de uma tela de criação de conta tradicional.

**Visão de Futuro (Account Manager Integrado):**
1. O jogador abre o OTClient e, na tela de login, tenta entrar sem preencher os campos de conta e senha (ou clica em um botão "Entrar" sem credenciais).
2. O cliente detecta a ausência de credenciais e, em vez de exibir um erro, envia uma requisição especial para o Login Server, que cria automaticamente uma nova conta com um nome único gerado (ex.: "MCR" + timestamp) e uma senha aleatória.
3. O Login Server também cria automaticamente o personagem "Alma" associado a essa conta e o posiciona no "Salão dos Destinos".
4. O jogador é automaticamente logado com esse personagem "Alma", onde é recebido pelo NPC Oráculo, que o guiará na criação de seu primeiro personagem real (nome, sexo, vocação).
5. Após a criação, o personagem "Alma" é desconectado e o jogador é redirecionado para a lista de personagens, onde seu novo herói aparece.

**Estado Atual (Protótipo para Testes):**
Atualmente, o fluxo automatizado não está implementado. Para testar a criação de personagem, um personagem "Alma" é criado manualmente no banco de dados para uma conta de teste específica. O jogador faz login com essa conta, seleciona "Alma" e inicia o diálogo com o Oráculo.

**Criação de Personagem via Oráculo (Fluxo Atual de Teste):**
1. O jogador entra no jogo com o personagem "Alma".
2. É automaticamente saudado pelo NPC "Oráculo dos Caminhos" no "Salão dos Destinos".
3. O Oráculo guia o jogador na escolha de um nome (com validação de caracteres permitidos), um sexo (masculino/feminino) e uma vocação (Guerreiro, Arqueiro, Mago, Druida).
4. Após confirmação, o servidor valida os dados, cria o novo personagem na tabela `players` e desconecta "Alma".
5. O jogador é redirecionado à tela de seleção de personagens, onde o novo herói aparece.

**Observação sobre o Botão "Criar Conta":**
O botão "Criar Conta" na tela de login é um placeholder temporário. Ele será removido quando o sistema de Account Manager estiver totalmente implementado.

2.7. Considerações de Segurança
Senhas: O projeto utiliza SHA1 para hash de senhas, garantindo compatibilidade imediata e simplificada entre Login Server e Canary. A robustez do sistema de criação de contas e login é a prioridade.

Conexões: Utilize sempre senhas fortes para o banco de dados e mantenha o software do servidor e do Login Server atualizados.

Firewall: Configure o firewall para permitir apenas as portas necessárias:
- 8080 (TCP) para o Login Server (acesso público HTTP).
- 7172 (TCP) para comunicação interna do Login Server com o Canary.
- 7173 (TCP) para o servidor de jogo (Canary), acesso público para os jogadores.
- 3306 (TCP) para o banco de dados (não deve ser exposto publicamente).

2.9. Criação de Personagem – "Salão dos Destinos" (Protótipo Funcional)

A criação de personagem no Projeto MCR é realizada através de uma experiência narrativa
imersiva denominada **"Salão dos Destinos"**. Atualmente em fase de protótipo funcional,
o sistema permite que novos jogadores criem seus heróis diretamente no cliente OTClient,
sem necessidade de ferramentas externas.

**Fluxo Atual (Protótipo):**
1. O jogador faz login com uma conta que contém o personagem **"Alma"** (criado
   manualmente para testes). **Atualmente, "Alma" não é um personagem dummy global;
   está acessível apenas na conta onde foi criado.**
2. Ao entrar no jogo com "Alma", o personagem aparece em uma área isolada do mapa
   ("Salão dos Destinos") e é saudado pelo NPC **"Oráculo dos Caminhos"**.
3. Através de um diálogo imersivo no chat, o Oráculo guia o jogador na escolha de:
   - Um **nome** (com validação de caracteres permitidos).
   - Um **sexo** (masculino ou feminino).
   - Uma **vocação** (Guerreiro, Arqueiro, Mago, Druida).
4. Após confirmação, o servidor valida os dados, cria o novo personagem na tabela `players`
   e desconecta o personagem "Alma".
5. O jogador é redirecionado à tela de seleção de personagens, onde o novo herói já
   aparece listado.

**Validação de Nomes (Encoding):**
Devido a limitações conhecidas do OTClient Redemption com a renderização de caracteres
acentuados em nomes de criaturas, o protótipo adota uma **lista branca de caracteres
permitidos**. São aceitos todos os caracteres ASCII imprimíveis (letras, números,
pontuação) e a maioria dos acentos comuns (ex.: á, é, ó, ç, â, ê, ô). Caracteres que
causam corrupção visual (ex.: ã, õ, í) são bloqueados com uma mensagem de erro clara.

**Limitações Atuais (Protótipo):**
- **O personagem "Alma" não é um dummy global.** Apenas a conta que o contém pode
  acessar o Salão dos Destinos.
- **O fluxo de criação automática de conta e personagem "Alma" ainda não foi implementado.**
- A lista branca de caracteres impede o uso de alguns acentos comuns em português.

**Visão de Futuro (Account Manager Completo):**
O NPC Oráculo, localizado no "Salão dos Destinos" e acessado através do personagem "Alma", será expandido para funcionar como um **"Administrador de Contas"** completo, oferecendo opções como:
- **Criar Personagem:** Fluxo atual (nome, sexo, vocação).
- **Alterar Senha:** Solicita senha atual e nova senha.
- **Deletar Personagem:** Lista personagens da conta e permite exclusão.
- **Recuperar Conta:** (Opcional) Gera um token de recuperação que pode ser usado em um comando no cliente.

**Próximos Passos:**
- Implementar a criação automática de conta e personagem "Alma" no primeiro acesso.
- Remover o botão "Criar Conta" da tela de login.
- Expandir o diálogo do Oráculo para incluir as funcionalidades administrativas.
- Padronizar o diálogo do Oráculo utilizando a biblioteca `npc_utils.lua`.

2.10. Criação e Posicionamento de NPCs no Mundo (Spawn)

O Canary utiliza o sistema Revscriptsys para scripts de NPC, que são arquivos .lua independentes e auto‑registráveis. No entanto, para que um NPC apareça no mapa, ainda é necessário declará-lo no arquivo de spawn do mundo. Esta seção documenta o processo completo e obrigatório para dar vida a um novo NPC.

2.10.1. Script do NPC (Revscriptsys)

O arquivo Lua do NPC deve ser colocado em data/npc/scripts/ e seguir a estrutura padrão (como canary.lua). Exemplo:

lua
local npcName = "MeuNovoNpc"
local npcType = Game.createNpcType(npcName)
local npcConfig = { name = npcName, /* ... configurações de outfit, loja, etc. ... */ }
-- ... lógica do NPC ...
npcType:register(npcConfig)
Nota: O nome definido em npcConfig.name DEVE corresponder exatamente ao atributo name no arquivo de spawn XML.

2.10.2. Arquivo de Spawn do Mundo (MCR-npc.xml)

O arquivo de spawn de NPCs está localizado em data/world/MCR-npc.xml. Ele contém uma lista de todos os NPCs que devem ser carregados automaticamente ao iniciar o servidor.

Estrutura do Arquivo:

xml
<?xml version="1.0"?>
<npcs>
    <!-- Exemplo de spawn para um NPC -->
    <npc centerx="666" centery="663" centerz="15" radius="6">
        <npc name="NomeExatoDoNPC" x="0" y="0" z="0" spawntime="60" />
    </npc>
</npcs>
Atributos Explicados:

<npcs>: Tag raiz obrigatória.

<npc> (container): Define uma área de spawn.

centerx, centery, centerz: Coordenadas do centro da área de spawn no mapa.

radius: Raio em tiles dentro do qual o NPC pode aparecer aleatoriamente (útil para criaturas, mas para NPCs geralmente usa-se radius="1" e coordenadas exatas).

<npc> (entidade): Define o NPC em si.

name: CRÍTICO. Deve ser idêntico ao npcConfig.name definido no script Lua.

x, y, z: Coordenadas relativas ao center do container. Para posicionamento exato, use x="0" y="0".

spawntime: Tempo em segundos para o NPC reaparecer se for removido (ex.: morto). Para NPCs, use um valor alto como 60 ou -1 para que nunca desapareça.

2.10.3. Procedimento Obrigatório para Adicionar um Novo NPC

Sempre que um novo NPC for criado, execute estes passos na ordem correta:

1. Crie o script Lua em data/npc/scripts/seu_npc.lua, definindo um npcConfig.name único e memorável.
2. Edite o arquivo data/world/MCR-npc.xml e adicione uma nova entrada <npc> com o name exato do script, posicionando-o nas coordenadas desejadas do mapa.
3. Reinicie o servidor Canary ou utilize o comando /reload npcs (se disponível) para aplicar as mudanças.

Checklist de Verificação:
- O nome no script Lua (npcConfig.name) corresponde exatamente (case‑sensitive) ao atributo name no XML?
- O arquivo Lua está salvo como UTF‑8 com BOM (para acentuação correta no servidor)?
- As coordenadas no XML (centerx, centery, centerz) são válidas e estão em um tile acessível do mapa?
- O servidor foi reiniciado após as alterações?

⚠️ Aviso de Segurança: Nunca delete o arquivo MCR-npc.xml nem o modifique sem backup. Uma tag mal formada pode impedir o carregamento de todos os NPCs do mundo.
⚠️ IMPORTANTE – XAMPP e MariaDB:
O XAMPP atualmente utiliza MariaDB em vez de MySQL puro. Para configurar a senha
do root, utilize o comando:
  SET PASSWORD FOR 'root'@'localhost' = PASSWORD('<Senha>'); 

Não utilize a sintaxe "ALTER USER ... IDENTIFIED WITH mysql_native_password",
que é específica do MySQL e não funciona no MariaDB.

2.12. Organização de Scripts Personalizados (Pasta MCR Scripts)

Para manter todos os scripts criados durante o desenvolvimento do Projeto MCR
separados dos arquivos originais do Canary, adotou-se uma pasta dedicada:
**`data/scripts/MCR Scripts/`** (caminho a partir da raiz do servidor Canary,
ex.: `E:\Projeto MCR\Canary\data\scripts\MCR Scripts\`).

**Motivo e Funcionamento:**
- O Canary utiliza o sistema **Revscriptsys**, que carrega automaticamente todo
  arquivo `.lua` dentro de `data/scripts/` e suas subpastas. Portanto, scripts
  colocados em `MCR Scripts` são reconhecidos pelo servidor sem configuração
  adicional.
- Scripts de NPC criados para o projeto devem ser salvos em
  `data/npc/scripts/MCR Scripts/` (ou diretamente em `data/npc/scripts/`,
  mas preferencialmente na subpasta para manter a organização).
- Arquivos de biblioteca ou utilitários (ex.: `npc_utils.lua`) devem ser
  referenciados por outros scripts através do caminho completo dentro do
  diretório `data/`, por exemplo:
  `dofile('data/scripts/MCR Scripts/npc_utils.lua')` ou
  `dofile('data/npc/scripts/MCR Scripts/npc_utils.lua')`, conforme o caso.

**Regra para Novos Arquivos:**
- **Sempre que possível**, todo novo arquivo `.lua` criado para o projeto
  (sistemas, bibliotecas, utilitários, scripts de criaturas, etc.) deve ser
  salvo em **`data/scripts/MCR Scripts/`**.
- Para **NPCs**, salve o script em **`data/npc/scripts/MCR Scripts/`**,
  mantendo a separação lógica. Lembre-se de adicionar a entrada correspondente
  no arquivo `data/world/MCR-npc.xml` (veja Seção 2.10).
- **Exceções:**
  * Modificações em arquivos já existentes do Canary ou do OTClient (ex.:
    `entergame.lua`, `config.lua`, módulos do cliente) devem permanecer em
    seus locais originais.
  * Scripts que dependem de caminhos relativos específicos definidos por
    sistemas do Canary (ex.: ações de movimento em `data/scripts/actions/`)
    podem precisar ficar em suas pastas originais. Nesse caso, o motivo será
    explicado na criação do script.

**Benefício:** Organização centralizada dos scripts do projeto, facilitando
localização, backup e versionamento.

⚠️ Atenção para PowerShell: Ao usar o terminal do Windows, caminhos com espaços devem
ser envolvidos entre aspas duplas. Exemplo:
  git add "docs/Projeto MCR.txt"
Nunca use barra invertida antes do espaço (estilo Bash), pois o PowerShell a interpreta
de maneira diferente.

Descrição: Fornece funções padronizadas para reconhecimento flexível de intenções,
personalização de pronomes/tratamento e variação de respostas.
--]]

NpcUtils = {}

-- Constantes de sexo (conforme definidas no Canary)
local PLAYERSEX_FEMALE = 1
local PLAYERSEX_MALE = 0

--[[
Verifica se a mensagem do jogador contém qualquer um dos sinônimos fornecidos.
A verificação é case‑insensitive e busca a palavra inteira para evitar falsos positivos.

@param mensagem string A mensagem digitada pelo jogador.
@param listaSinonimos table Uma tabela de strings contendo os sinônimos.
@return boolean True se a mensagem corresponder a pelo menos um sinônimo.
]]
function NpcUtils.correspondeAcao(mensagem, listaSinonimos)
mensagem = mensagem:lower()
for _, sinonimo in ipairs(listaSinonimos) do
-- Padrão que busca a palavra inteira, cercada por espaços, pontuação ou início/fim da string
local padrao = '[%p%c%z%s]' .. sinonimo:lower() .. '[%p%c%z%s]'
if mensagem:find(padrao) or mensagem:find('^' .. sinonimo:lower() .. '$') or mensagem:find('^' .. sinonimo:lower() .. '[%p%c%z%s]') or mensagem:find('[%p%c%z%s]' .. sinonimo:lower() .. '$') then
return true
end
end
return false
end

--[[
Retorna uma tabela com pronomes, artigos e vocativos personalizados de acordo com
o sexo e a vocação do jogador. Essencial para uma narrativa imersiva e inclusiva.

@param player userdata O objeto Player do jogador.
@return table Tabela com os campos: artigo, pronome, possessivo, tratamento, bemVindo, sufixoVoc, vocativo.
]]
function NpcUtils.getTratamento(player)
local sexo = player:getSex()
local vocacao = player:getVocation():getName()
local tr = {}

if sexo == PLAYERSEX_FEMALE then
tr.artigo = "a"; tr.pronome = "ela"; tr.possessivo = "sua"
tr.tratamento = "senhorita"; tr.bemVindo = "bem‑vinda"; tr.sufixoVoc = "a"
else
tr.artigo = "o"; tr.pronome = "ele"; tr.possessivo = "seu"
tr.tratamento = "senhor"; tr.bemVindo = "bem‑vindo"; tr.sufixoVoc = ""
end

-- Vocativos fantásticos de acordo com a vocação (padrão MCR)
local vocativos = {
Knight = "guardião" .. tr.sufixoVoc .. " de armadura reluzente",
Paladin = "atirador" .. tr.sufixoVoc .. " de flechas sagradas",
Sorcerer = "tecelão" .. tr.sufixoVoc .. " dos véus arcanos",
Druid = "guardiã" .. tr.sufixoVoc .. " dos ciclos naturais"
}
tr.vocativo = vocativos[vocacao] or "aventureir" .. tr.sufixoVoc .. " dos reinos distantes"

return tr
end

--[[
Seleciona aleatoriamente uma frase de uma lista e substitui o placeholder %s
pelo nome do jogador, garantindo variação nas interações.

@param listaFrases table Lista de strings com placeholders %s.
@param player userdata O jogador para o qual a frase será personalizada.

function NpcUtils.escolherFraseVariada(listaFrases, player)

return frase:format(nome)
end

SISTEMA DE QUESTS HÍBRIDO (SQH) – CHAT IMERSIVO COM PALAVRAS DESTACADAS
Escopo: Evolução do antigo SQT para um modelo que combina narrativa rica no chat com interatividade moderna (palavras destacadas, HUD persistente, notificações toast).

5.1. Visão Geral do Subsistema
Este subsistema redefine a interação de missões no Projeto MCR. O chat continua sendo o palco principal da narrativa, garantindo que o jogador leia e se conecte com os NPCs. Entretanto, para eliminar a fricção de ter que adivinhar palavras‑chave, as ações mais importantes são destacadas com cores no texto. Além disso, um HUD de Missões (Quest Tracker) fornece um lembrete visual constante do progresso, e notificações toast entregam feedback imediato.

Objetivo Principal: Oferecer uma experiência narrativa profunda, sem sacrificar a conveniência que os jogadores modernos esperam.

5.2. Convenção de Marcação: Tags de Cor {{#RRGGBB}}texto{{/}}
Para diferenciar os elementos interativos do texto narrativo, utilizamos as tags de cor nativas do Canary/OTClient. O cliente exibirá o texto dentro das tags na cor especificada. O jogador ainda precisa digitar a palavra, mas a cor serve como dica visual clara.

Categoria Formato Exemplo Visual no Chat Ação Esperada
Ação (Verbo) {{#00FF00}}verbo{{/}} (verde) {{#00FF00}}aceitar{{/}} Jogador deve digitar "aceitar".
Item de Quest {{#00BFFF}}quantidade Nome do Item{{/}} (azul) {{#00BFFF}}10 Peles de Lobo Etéreo{{/}} Apenas destaque visual.
Local/NPC Alvo {{#FFD700}}Nome do Local{{/}} (dourado) {{#FFD700}}Torre do Astrônomo Perdido{{/}} Apenas destaque visual.

Nota: Esta abordagem não exige nenhuma modificação no cliente OTClient, utilizando recursos nativos do protocolo.

5.3. Fluxograma de Interação Híbrida
Este é o fluxo padrão que todo NPC de quest deve seguir. Proíbe‑se o uso de player:showModalWindow() para progresso de missão.

Saudação Contextual: O NPC identifica o jogador e seu estado de missão.
Exemplo: "Ah, [Jogador]! Vejo que retornou. Conseguiu encontrar as {{#00BFFF}}10 Peles de Lobo Etéreo{{/}}?"

Reconhecimento de Intenção: O script escuta palavras‑chave específicas para disparar ações.
Exemplo: Se o jogador digitar missão, tarefa, trabalho.
Resposta do NPC: "Preciso que você vá até o {{#FFD700}}Lago dos Reflexos{{/}} e me traga {{#00BFFF}}5 Lágrimas de Sereia{{/}}. O que me diz? {{#00FF00}}aceitar{{/}}?"

Confirmação via Digitação: O jogador digita "aceitar" (ou sinônimo configurado).

Ação do Servidor: Seta storage de missão ativa e:

Envia um Extended Opcode para atualizar o HUD de Missões.

Opcionalmente, dispara uma notificação toast: "Missão iniciada: Lágrimas de Sereia".

Acompanhamento pelo HUD: Durante a missão, o jogador pode ver o progresso no canto da tela:
[Missão] Lágrimas de Sereia

Coletar Lágrimas de Sereia: 2/5
O HUD é atualizado automaticamente via Opcodes quando o jogador coleta os itens.

Entrega/Progresso: O jogador retorna ao NPC e digita "entregar".

Validação do Servidor: Checa itens, remove, concede recompensa.

Sucesso: NPC envia mensagem narrativa com destaque para a recompensa:
"Incrível! Tome seus {{#FFD700}}100 Fragmentos de Prata Lunar{{/}} e este {{#00BFFF}}Talismã da Maré{{/}}."
O servidor envia Opcodes para limpar o HUD da missão e exibir uma notificação toast de conclusão.

5.5. Diretrizes de Qualidade para Quests Híbridas

Proibição de Janelas Modais: O NPC de quest nunca deve chamar player:showModalWindow() para decisões de missão. Exceção: Loja, Banco e confirmações críticas.

Uso Consistente de Tags de Cor: Termos acionáveis DEVEM ser destacados com {{#00FF00}} (verde). Itens com {{#00BFFF}} (azul). Locais com {{#FFD700}} (dourado).

HUD Sempre Visível: O Quest Tracker deve aparecer assim que o jogador tiver uma missão ativa e desaparecer ao concluí‑la.

Feedback Imediato: Toda mudança de estado da missão deve gerar uma notificação toast e/ou som sutil.

Respeito à Narrativa: Mesmo com destaques, o diálogo do NPC deve permanecer rico e variado.

5.6. Lista de Verificação de Implementação (Checklist SQH)
Antes de marcar uma quest como "Concluída", verifique:

Os diálogos do NPC usam tags de cor para termos acionáveis: {{#00FF00}}aceitar{{/}}, {{#00FF00}}entregar{{/}}.

Não há chamadas para player:showModalWindow() no fluxo da missão.

Ao aceitar a missão, um Opcode é enviado para exibir/atualizar o HUD de Missões.

Ao coletar itens, o progresso no HUD é atualizado em tempo real.

Ao concluir a missão, o HUD é ocultado e uma notificação toast é exibida.

A recompensa é entregue diretamente no inventário e anunciada no chat com destaque.

6.2. Padrão de Codificação e Comentários

Idioma dos Comentários: Português do Brasil (pt‑BR). Todos os comentários em arquivos .cpp, .h e .lua devem ser escritos em português claro e correto.

Cabeçalho de Arquivo: Todo novo arquivo Lua ou módulo deve conter um cabeçalho descritivo:
lua

Nomes de Funções e Variáveis: Utilize camelCase para variáveis e funções locais, e PascalCase para funções globais/bibliotecas. Exemplo: local playerName, function QuestSystem.updateProgress().

Evite Código Mágico: Números "mágicos" (valores literais sem explicação) são proibidos. Atribua‑os a constantes nomeadas com comentários. Exemplo: local MAX_PLAYER_LEVEL = 500.

6.3. Segurança e Validação de Dados

Sanitização de Entrada do Jogador: Ao processar texto digitado pelo jogador (ex.: em onCreatureSay), sempre converta para minúsculas (msg:lower()) e remova espaços extras antes de aplicar find ou comparações.

Validação de Dados do Cliente: Nunca confie cegamente nos dados recebidos do cliente. Sempre valide parâmetros de Extended Opcodes no lado do servidor. Verifique se os IDs recebidos existem, se as quantidades são válidas e se o jogador tem permissão para a ação.

Prevenção de Injeção de SQL: Embora o Canary utilize consultas parametrizadas, ao escrever queries manuais (ex.: db.query), nunca concatene strings diretamente. Use sempre placeholders (%s, %d).

Proteção de Loops Infinitos: Em loops while ou repeat, garanta que haja uma condição de saída clara e um limite máximo de iterações para evitar travamentos do servidor.

6.4. Performance e Otimização

Extended Opcodes: O tamanho máximo de um pacote Extended Opcode é de 65KB. Evite enviar grandes volumes de dados de uma só vez. Para dados extensos, utilize múltiplos pacotes ou um sistema de paginação.

Acesso a Storage: Evite chamadas excessivas a player:getStorageValue() dentro de loops. Armazene o valor em uma variável local sempre que possível.

Iteração sobre Jogadores: Ao iterar sobre Game.getPlayers(), esteja ciente de que isso pode ser custoso em servidores com muitos jogadores online. Utilize eventos específicos ou sistemas de cache quando aplicável.

6.5. Consistência de Design e Narrativa (Tema Fantasia)

Tom de Voz dos NPCs: Mantenha a coerência com o mundo fantasioso. Um mercador anão deve ter um tom diferente de um elfo ancião. Consulte a biblioteca npc_utils.lua para funções de personalização.

Uso de Janelas Modais: Estritamente limitado a situações críticas que demandam confirmação explícita e não devem ser ignoradas (ex.: reset de atributos, ações destrutivas irreversíveis), além de Loja e Banco. Jamais utilize janelas modais para aceitar ou entregar missões.

Destaque de Texto: Utilize tags de cor {{#RRGGBB}}texto{{/}} para destacar termos interativos e itens importantes. Verde (#00FF00) para ações, azul (#00BFFF) para itens, dourado (#FFD700) para locais.

6.6. Gerenciamento de Armazenamento (Storages e KVPs)

Faixas de Storage: Para evitar conflitos, defina e documente faixas exclusivas para cada sistema do MCR. Sugestão inicial:

50000 – 50999: Missões e Progresso de História.

51000 – 51999: Habilidades Especiais e Talentos.

52000 – 52999: Configurações do Jogador (UI, preferências).

Dados Complexos: Para armazenar tabelas ou múltiplos valores, utilize json.encode() e json.decode() em uma única storage, ao invés de múltiplas storages separadas.

Limpeza: Crie scripts de limpeza (/limparstorages) para remover dados de teste e permitir que jogadores reiniciem missões, se for uma funcionalidade desejada.
Faixas de Extended Opcodes (Comunicação Cliente‑Servidor):
Para evitar conflitos com opcodes nativos do Canary e de outras customizações, todas as comunicações customizadas do Projeto MCR devem utilizar opcodes na faixa de 160 a 255.

160 – 179: Reservado para sistema de criação de personagem ("Salão dos Destinos").

180 – 199: Reservado para o Sistema de Quests Híbrido (SQH) – atualizações de HUD, notificações toast.

200 – 219: Reservado para o futuro "Administrador de Contas" (Account Manager "Alma").

220 – 255: Livre para outros sistemas.
Registro: Todo opcode utilizado deve ser documentado no MCR_DevLog.txt e na seção de Opcodes deste documento.

6.7. Testes e Depuração

Modo Debug: Inclua uma variável global MCR_DEBUG = true no servidor que, quando ativada, imprime logs detalhados no console sobre o funcionamento dos sistemas.

Comandos de Admin: Para cada sistema complexo (ex.: missões), crie comandos de administrador (ex.: /setquestprogress [id] [valor]) para facilitar os testes.

Tratamento de Erros: Utilize pcall em chamadas de json.encode/decode para capturar erros de formatação e evitar quebras no script.

Ao criar qualquer novo conteúdo (NPC, missão, sistema) para o mundo de fantasia do MCR, siga este fluxo de trabalho obrigatório:

6.11. Diretrizes para APIs de Autenticação (Login Server)
Para suportar a criação automática de contas e o gerenciamento de personagens diretamente do OTClient, o Login Server expõe os seguintes endpoints REST.

Endpoint Método Descrição Validações Obrigatórias e Comportamento Esperado
/register POST Cria uma nova conta de jogador. Entrada (JSON): { "account": "nomeUsuario", "password": "senha123" }
Validações:

account: 3 a 32 caracteres alfanuméricos. Deve ser único na tabela accounts.

password: Mínimo de 8 caracteres, contendo pelo menos uma letra e um número.
Processamento:

Validar campos conforme acima.

Gerar hash SHA1 da senha.

Inserir registro na tabela accounts com password = hash, email = "" (string vazia).

Retornar status HTTP 200 em caso de sucesso, ou 400/409 com mensagem de erro descritiva em pt‑BR.
Exemplo de resposta de sucesso: {"status":"success","message":"Conta criada com sucesso!"}
Exemplo de erro: {"status":"error","message":"O nome de conta já está em uso."}
/login POST Autentica um jogador. Entrada (JSON): { "account": "nomeUsuario", "password": "senha123" }
Processamento:

Consultar a conta no banco de dados e verificar o hash SHA1.

Se autenticado, retornar lista de personagens e informações do servidor de jogo.
/change_password POST Altera a senha da conta (via Account Manager "Alma"). Entrada (JSON): { "account": "nomeUsuario", "current_password": "senhaAtual", "new_password": "novaSenha123" }
Validações:

Autenticação via senha atual.

new_password atende aos critérios de força.
Processamento:

Autenticar.

Atualizar password com hash SHA1 da nova senha.

Retornar sucesso.

Observações de Segurança e Implementação:

Todos os endpoints devem operar sobre HTTPS em ambiente de produção.

As respostas devem incluir cabeçalhos CORS apropriados para permitir requisições originadas do OTClient.

O hash de senha utilizado é SHA1, garantindo máxima compatibilidade entre Login Server e Canary.

A tabela accounts utilizada pelo Login Server deve ser a mesma do Canary (BancoServer.accounts). O campo email é definido como string vazia.

Para garantir a exibição correta de mensagens acentuadas no OTClient sem depender da codificação do backend, adote o padrão de códigos de erro (error_code). O servidor retorna um código padronizado (ex.: "account_name_taken"), e o cliente mapeia esse código para uma string em português definida localmente no script Lua (salvo como ISO‑8859‑1).

Configuração do phpMyAdmin após definir senha no MySQL/MariaDB:
  $cfg['Servers'][$i]['password'] = '<Senha>'; (não informada por questão de segurança, arquivo com acesso publico no Git)
  $cfg['Servers'][$i]['auth_type'] = 'config';

  para obter o conteúdo e compreender sua estrutura.
- Se o conteúdo RAW conflitar com o que o usuário está solicitando ou descrevendo,
  o assistente deve assumir que a versão RAW está desatualizada e basear-se nas
  instruções mais recentes da conversa e na documentação do projeto.
- Caso identifique uma discrepância relevante, deve reportá-la educadamente,
  perguntando se deseja atualizar o repositório, mas sem rejeitar automaticamente
  a solicitação original.

  6.14. Melhoria Contínua da Documentação e Aprendizado com Erros

⚠️ DEVER PROATIVO DO ASSISTENTE

O assistente não deve ser um mero resolvedor de tarefas pontuais. Ele é corresponsável
pela evolução da base de conhecimento do Projeto MCR. Para tanto, ao longo de cada
interação, o assistente DEVE:

1. **Monitorar os próprios erros:** Se um comando falhou, um caminho foi mal interpretado,
   uma sugestão se mostrou inviável ou um problema se repetiu mais de uma vez, isso é um
   sinal claro de que a documentação pode estar incompleta ou de que o assistente está
   preso em um padrão ineficaz.

2. **Investigar a causa raiz:** Em vez de apenas aplicar uma correção superficial, o
   assistente deve se perguntar: *"Por que isso aconteceu? Faltou informação em qual
   documento? O Projeto MCR.txt está omisso? O DevLog registra essa armadilha? As
   Instruções do Assistente deixam alguma margem para dúvida?"*

3. **Sugerir atualizações documentais:** Sempre que um aprendizado relevante for
   identificado — uma nova lição técnica, um erro frequente, uma melhoria de fluxo —
   o assistente DEVE propor explicitamente:
   - Adições ou correções no **Projeto MCR.txt**.
   - Uma nova entrada no **MCR_DevLog.txt** documentando o ocorrido.
   - Ajustes nas **MCR_INSTRUCOES_ASSISTENTE.txt** para evitar recorrências.

4. **Evitar loops de ineficácia:** Se o mesmo tipo de problema aparece repetidamente
   (ex.: comandos Git com sintaxe incorreta, acentuação quebrada, confusão de portas),
   o assistente deve reconhecer que a solução não está sendo efetiva e automaticamente
   revisitar os pontos acima, em vez de continuar aplicando remendos.

Exemplo concreto: na Fase 36, descobriu-se que comandos Git com barra invertida falham
no PowerShell. O assistente aprendeu a usar aspas duplas e propôs a adição dessa nota
na seção 2.11.3 do Projeto MCR.txt e nas Instruções do Assistente. Esse ciclo
virtuoso deve se repetir para qualquer aprendizado significativo.

Resultado esperado: documentação cada vez mais robusta, assistente cada vez mais
eficiente e prevenção ativa de erros já conhecidos.
