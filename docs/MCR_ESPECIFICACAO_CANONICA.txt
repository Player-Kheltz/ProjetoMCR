PARTE 1: CONTEXTO E FILOSOFIA (A "ALMA" DO PROJETO)
================================================================================

1.1. Identidade e Tema Central
- Nome do Projeto: Projeto MCR
- Tema: "Uma Aventura Fantasiosa". Um universo de fantasia original e independente,
  sem qualquer ligação com o lore do Tibia oficial. A criatividade é o limite.
- Idioma Oficial: Português do Brasil (pt-BR). Aplicado a TODO texto visível ao
  jogador, mensagens de sistema, comentários de código e documentação.

1.2. O Pilar Inegociável: Jornada 100% no Cliente
Nenhuma experiência essencial do jogador deve ocorrer fora do OTClient. Criação de
conta, login, seleção/criação de personagem, quests, loja, gerenciamento de conta...
TUDO deve ser resolvido dentro do cliente. O site (MyAAC) é um complemento opcional,
jamais um requisito.

1.3. Escopo de Customização: Alcance Máximo (REGRA DE OURO)
O Projeto MCR não se limita a scripts. Todos os componentes do ecossistema são
passíveis de modificação em nível de código-fonte para atingir os objetivos do projeto.
Isso inclui:
- Servidor Canary (C++ / Lua): Modificações no núcleo, novas bindings, sistemas.
- OTClient Redemption (C++ / Lua): Novas interfaces, módulos, renderização customizada.
- Login Server (Go): Novos endpoints REST, lógica de autenticação estendida.
- Map Editor: Ajustes para compatibilidade com o protocolo.

Regra de Ação: Se uma funcionalidade não puder ser feita via script, modifique o fonte.

================================================================================
PARTE 2: ESPECIFICAÇÃO TÉCNICA E ARQUITETURA (O "CORPO" DO PROJETO)
================================================================================

2.1. Componentes do Ecossistema (Versões Oficiais)
- Servidor de Jogo: opentibiabr/canary (Tag v3.4.1 - Protocolo 15.00)
- Cliente: opentibiabr/otclient (Redemption - Protocolo 15.00)
- Servidor de Login: opentibiabr/login-server
- Banco de Dados: MySQL (Schema: "BancoServer")

2.2. Estado Atual da Implementação (Validado e Funcional)
- ✅ Registro de Conta: 100% funcional no cliente via `registro.lua`. Interface
  polida (360x360) com validação, navegação por teclado e feedback visual.
- ✅ Login: Realizado via HTTP POST do OTClient para o Login Server. Lista de
  personagens exibida corretamente.
- ✅ Autenticação: Login Server traduz nome de conta para e-mail. Senhas utilizam
  hash SHA1 para compatibilidade Canary-LoginServer.
- ✅ Entrada no Jogo: Validada. Comunicação OTClient ↔ Login Server ↔ Canary
  operante e estável.
- ⚠️ Criação de Personagem: Protótipo funcional ("Salão dos Destinos"). O fluxo é
  iniciado entrando com um personagem dummy ("Alma"), que é guiado por um NPC
  (Oráculo) para criar um novo herói. Ainda não há botão "Criar Personagem" na
  interface de seleção.

2.3. Configurações Críticas (NÃO ALTERAR SEM ANÁLISE)
- Portas Canary: Status=7171, Login=7172, Game=7173.
- Login Server: Porta HTTP=8080. Variável de ambiente `SERVER_PORT` no `.env` DEVE
  ser `7173` (aponta para o ProtocolGame do Canary).
- OTClient: Configurado com `httpLogin = true`. Cache local configurado para pasta
  `./cache/` (portátil).
- Codificação de Arquivos:
  - Scripts Lua (Servidor): UTF-8 com BOM.
  - Scripts Lua (Cliente - carregados via dofile): ISO-8859-1.
  - Backend Go (Login Server): UTF-8 com BOM. Mensagens de erro devem usar `error_code`.

PARTE 3: MANUAL DE DESIGN E INTERAÇÃO (A "EXPERIÊNCIA")
================================================================================

3.1. Hierarquia de Ferramentas de Interface (Obrigatório)
1.  Chat Narrativo com [Links Clicáveis]: Para progresso de quests, diálogos,
    interações com NPCs. (Foco na imersão).
2.  HUD Persistente: Para rastreamento de missões (Quest Tracker) e informações
    contínuas. (Feedback passivo).
3.  Notificações Toast: Para feedback de ações concluídas. (Feedback efêmero).
4.  Janelas Modais: **USO RESTRITO E EXCEPCIONAL**. Apenas para Loja, Banco e
    confirmações CRÍTICAS e IRREVERSÍVEIS. **PROIBIDO** para progresso de missão.

3.2. Sistema de Diálogo (NPC Utils)
- Biblioteca Obrigatória: `data/npc/lib/npc_utils.lua`.
- Funções Padrão:
  - `NpcUtils.correspondeAcao(msg, listaSinonimos)`: Para reconhecer intenções.
  - `NpcUtils.getTratamento(player)`: Para personalizar por sexo/vocação.
  - `NpcUtils.escolherFraseVariada(listaFrases, player)`: Para evitar repetição.
- Links Clicáveis: Use colchetes `[aceitar]`. O cliente deve ser customizado para
  renderizá-los como hyperlinks que enviam o texto ao chat.

3.3. Sistema de Quests Híbrido (SQH)
- Fluxo Padrão: NPC fala com links -> Jogador clica ou digita -> Servidor processa ->
  Servidor envia Opcode para atualizar HUD/Toast.
- Opcodes Reservados: Faixa **160-255**.
  - 160-179: Salão dos Destinos.
  - 180-199: SQH (HUD e Toasts).
  - 200-219: Futuro Administrador de Contas.

4.1. Protocolo de Resposta (Aplicar em TODA interação)
1.  **Idioma:** Responder sempre em pt-BR, com gramática e acentuação impecáveis.
2.  **Verificação de Código:** Antes de sugerir código, validar IDs, funções e
    parâmetros. Se não for possível confirmar, adicionar `-- VERIFICAÇÃO PENDENTE`.
3.  **Foco na Jornada 100% Cliente:** Nunca sugerir uma solução que force o jogador
    a sair do OTClient para uma ação essencial.
4.  **Proibição de Janelas Modais para Quest:** Revisar qualquer sugestão de código
    que utilize `player:showModalWindow()` para progresso narrativo.
5.  **Sincronização com DevLog:** Se a resposta envolver uma correção de bug,
    descoberta técnica ou mudança de abordagem, **RECOMENDAR EXPLICITAMENTE** a adição
    de uma nova entrada no `MCR_DevLog.txt`, sugerindo o título e o conteúdo.
