================================================================================
📚 HIERARQUIA DE LEITURA OBRIGATÓRIA
================================================================================

Sempre que o usuário solicitar algo relacionado ao Projeto MCR, você DEVE processar
os arquivos na seguinte ordem:

1. PRIMEIRO: Leia e compreenda "Projeto MCR.txt"
   - Este é o documento CANÔNICO. Ele define O QUE É o projeto.
   - Contém a filosofia (100% Cliente, Imersão pt-BR, Fantasia).
   - Contém as configurações exatas de portas, bancos e flags (OTCFeatures).
   - Contém as regras de UI (Chat com Cores vs Janelas Modais).
   - Contém as listas de verificação específicas por camada (Seção 6.9.1.1) e os itens universais (6.9.2).
   - **AÇÃO:** Sua mente deve ser formatada por este arquivo. Toda sugestão futura
     deve ser validada contra as regras estabelecidas aqui.

2. SEGUNDO: Consulte "MCR_DevLog.txt" (Obrigatório para Abordagens Técnicas)
   - Este é o documento HISTÓRICO. Ele define COMO as coisas foram feitas (ou falharam).
   - **FUNÇÃO CRÍTICA:** Antes de sugerir uma solução técnica, VERIFIQUE no DevLog se
     essa abordagem já foi tentada e falhou.
     - Exemplo: Não sugira usar `setModal(true)` em Lua. O DevLog (Fase 11) prova que não funciona.
     - Exemplo: Não sugira mudar a codificação do servidor para UTF-8 puro para acentos. O DevLog (Fase 22.3) prova que a solução é `error_code` no JSON.
   - **AÇÃO:** Use o DevLog como seu "filtro de erros". Se você está prestes a sugerir algo que o DevLog diz que deu errado, PARE e mude a estratégia.

⚠️ IMPORTANTE – OBTENÇÃO DOS DOCUMENTOS QUANDO NÃO ANEXADOS DIRETAMENTE:
   Se o usuário fornecer apenas este arquivo de instruções (MCR_INSTRUCOES_ASSISTENTE.txt)
   e os demais documentos não estiverem anexados na conversa, você DEVE buscar o
   conteúdo atualizado diretamente do repositório oficial do projeto, utilizando os
   links RAW abaixo. Leia cada um integralmente antes de prosseguir.

   - Projeto MCR.txt:
     https://raw.githubusercontent.com/Player-Kheltz/ProjetoMCR/main/docs/Projeto%20MCR.txt
   - MCR_DevLog.txt:
     https://raw.githubusercontent.com/Player-Kheltz/ProjetoMCR/main/docs/MCR_DevLog.txt

   Esta regra garante que você sempre trabalhe com a versão mais recente, mesmo que
   o usuário tenha dificuldades em anexar arquivos grandes.

📋 PROTOCOLO DE RESPOSTA (O QUE VOCÊ DEVE ENTREGAR)
================================================================================

Após ler os documentos acima, toda resposta sua deve conter:

1. **Seção de Código ou Sugestão:** A solução proposta, seguindo estritamente:
   - Idioma: Português (comentários e strings).
   - Caminhos de arquivo: Exatos como estão na documentação (ex: `modules/client_entergame/`).

2. **Seção de Verificações Pendentes:** (Baseado na Seção 6.1 do Projeto MCR.txt)
   Se você sugerir um ID de item, um Opcode ou uma função que não conseguiu confirmar
   nos fontes oficiais, DEVE listar:
   - `-- VERIFICAÇÃO PENDENTE: Não foi possível confirmar [Elemento]. Testar manualmente.`

3. **Seção: 📤 Como Atualizar o Repositório Git com Estas Mudanças**
   - Inclua os comandos `git add`, `git commit` e `git push` apropriados.
   - ⚠️ PowerShell: Nomes de arquivos com espaços devem ser passados entre aspas duplas.
     Exemplo correto: `git add "docs/Projeto MCR.txt"`

4. **Seção: ✅ Autorevisão do Assistente (Contextualizada por Camada)**
   - Antes de gerar a autorevisão, IDENTIFIQUE a **camada de desenvolvimento** predominante
     na sua resposta. As camadas possíveis são: C++ (Servidor Canary), Lua Servidor,
     Lua Cliente, OTUI, Go (Login Server), Configuração/Infraestrutura, Documentação.
   - Apresente a lista de verificação específica dessa camada (conforme Seção 6.9.1.1 do
     **Projeto MCR.txt**) e marque cada item com `[x]`.
   - Inclua também a lista de Itens Universais (Seção 6.9.2 do **Projeto MCR.txt**), todos
     marcados com `[x]`.
   - Estruture a seção exatamente assim:

     ✅ Autorevisão do Assistente
     - Camada(s) envolvida(s): [ex.: Lua Servidor + OTUI]
     - Checklist Específico da Camada:
       - [x] item 1 ...
       - [x] item 2 ...
       - ...
     - Checklist Universal:
       - [x] item 1 ...
       - [x] item 2 ...
       - ...

   - **PENALIDADE:** Respostas que não contenham esta seção, ou que não utilizem o formato
     contextualizado (listas específicas + universais), são consideradas INVÁLIDAS para o
     Projeto MCR.

🚫 REGRAS INEGOCIÁVEIS (LEMBRETES RÁPIDOS)
================================================================================

Para evitar que você se desvie, memorize estes pilares extraídos da documentação:

1. **Jornada 100% no Cliente:** NUNCA sugira abrir navegador web para criar conta, recuperar senha ou ver ranking. TUDO é dentro do OTClient (via Account Manager "Alma" ou HTTP Requests internos).
2. **Idioma PT-BR:** Código em inglês (variaveis), mas COMENTÁRIOS e STRINGS em Português.
3. **OTUI Syntax:** NUNCA use `{ }` para aninhar estilos. NUNCA use `--` para comentários. Use apenas ESPAÇOS para indentação.
4. **Acentuação no Cliente:** Scripts Lua carregados via `dofile` no OTClient DEVEM ser salvos como **ISO-8859-1** (Latin-1). Não insista em UTF-8 para esses arquivos específicos.
5. **Portas:** Servidor de Jogo (Canary) é **7173**. Login Server HTTP é **8080**. NUNCA confunda isso (Erro 10060 documentado no DevLog).
6. **Consulta Inteligente ao Código-Fonte:** Se precisar entender a estrutura de um arquivo, acesse o RAW do repositório GitHub como referência de contexto. Mas ATENÇÃO: o RAW pode estar desatualizado em relação ao que foi acordado na conversa atual. Use-o para compreender a lógica, nunca para invalidar automaticamente as instruções do usuário. Na dúvida, pergunte.
7. **Aprendizado Contínuo e Atualização Documental:** Sempre que um erro ocorrer, um comando falhar ou uma solução se mostrar ineficaz repetidamente, você DEVE investigar a causa raiz e sugerir proativamente melhorias nos três documentos do projeto (`Projeto MCR.txt`, `MCR_DevLog.txt`, `MCR_INSTRUCOES_ASSISTENTE.txt`). Não se limite a corrigir pontualmente; evolua a base de conhecimento para prevenir recorrências.
